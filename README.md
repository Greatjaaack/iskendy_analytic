# Iskendy Analytics

Система финансовой аналитики и учёта костов для ресторана «Искенди».
Тянет продажи из iiko, ведёт в собственной БД поставщиков, цены, тех-карты (ТТК) и
себестоимость блюд. Цель — всегда видеть косты и автоматизировать заказ продуктов.

> Точка: **Искенди**, `storeId = 161059`, iiko domain `iskendi`.

---

## Что умеет

- **Дашборд** (продажи из iiko): KPI с дельтами, выручка/часы по каналам обслуживания,
  аналитика состава и наполненности чека, продажи блюд с с/с и маржой, погода.
  Период — Сегодня / Неделя / Месяц **или произвольный диапазон**. Подробно по каждому
  блоку — ниже в разделе [«Дашборд — что показывает каждый блок»](#дашборд--что-показывает-каждый-блок).
- **Поставщики** — список с выгрузкой в Excel; карточка с редактированием реквизитов,
  **несколькими контактами и каналами связи** (телефон/WhatsApp/Telegram/email, с
  валидацией), товарами с ценами и **торговой маркой**, файлами (накладные/прайсы).
- **Номенклатура ↔ ТТК** — каталог ингредиентов, тех-карты с составом и себестоимостью
  (в т.ч. двухуровневые блюдо → полуфабрикат → сырьё) + вкладка **«Блюда ↔ ТТК»**:
  привязка проданных блюд к тех-картам (даёт с/с по блюду, когда POS-имя ≠ имя ТТК).
- **Импорт** из Excel «ТТК и матрица продуктов» — наполняет поставщиков, ингредиенты,
  цены и ТТК одним действием (привязки блюд и торговые марки переживают реимпорт).
- **Светлая/тёмная тема**.
- _В разработке_: накладные (косты), остатки из iiko, автозаказ. См. `PLAN.md`.

> Бóльшая часть метрик тянется из iiko **вживую** на каждый запрос (выручка/часы по
> каналам, блюда, состав/наполненность чека — через OLAP), поэтому при смене периода
> возможна задержка ~1–3 c. Выручка по дням (режим «Сумма») — из локальной БД (синк
> планировщиком).

## Дашборд — что показывает каждый блок

Период (Сегодня / Неделя / Месяц / произвольный диапазон) задаётся сверху и применяется
ко всем блокам. **Месяц = с 1-го числа текущего месяца по сегодня** (month-to-date), а не
последние 30 дней. Неделя = последние 7 дней. «Канал» = тип обслуживания: **в зале / с собой / доставка** — берётся из
модификатора категории «Статус» заказа, а доставка — ещё и из меню-категории «Доставка»
(подробности в `IIKO_WEB_API.md`).

Дашборд разбит на **3 вкладки** (KPI-карточки — всегда сверху, над вкладками):
- **Пульс** (деньги и динамика): Выручка по дням · Чеки по типу обслуживания.
- **Операции** (когда и как нагружена точка): Свод по дням недели · Выручка по часам · Наполненность чеков · Продажи по часам.
- **Меню** (что и насколько выгодно): Состав чека · Что продаётся по статусам · Меню-инжиниринг · Продажи блюд.

Раскладка вкладок задаётся в `frontend/src/pages/Dashboard.tsx` (стейт `tab`).

| Блок | Источник (API) | Что показывает | Управление |
|---|---|---|---|
| **KPI-карточки** | `/api/revenue`, `/api/revenue/kpi-by-channel` | Выручка, Средний чек, Чеков + **▲▼ % к прошлому аналогичному периоду**. Под каждым — разбивка **🛵 доставка / 🏠 не доставка** (брутто-суммы по каналам). | — |
| **Шапка «Сегодня»** | `/api/revenue` | Только при периоде «Сегодня»: дата, день недели, погода в Москве. | — |
| **Выручка по дням** | `/api/revenue`, `/api/revenue/by-channel` | Выручка/средний чек/чеки по дням. Под датой на оси X — погода и **±° к тому же дню прошлого периода**. В режиме «Сумма» — пунктирная **опорная линия среднесуточной выручки** за период (видно дни выше/ниже нормы). | Режимы **Сумма** (выручка+ср.чек+чеки) / **По статусам** (стек каналов с чипами вкл/выкл) / **Погода** (выручка тек. vs прош. период + температуры). Тип графика (Область/Линия/Ступени/Столбцы). Фильтр дней недели (мультивыбор, Будни/Выходные) — **скрыт, когда выбран один день**. |
| **Свод по дням недели** | `/api/revenue/by-weekday` | Таблица-агрегат: для каждого дня недели (Пн…Вс) за период — сколько таких дней попало, суммарная и средняя за день выручка, чеки, средний чек. Видно, какие дни недели «вытягивают». В колонке «Ср. за день» — **встроенные мини-бары** (ударный/провальный день виден глазами). | **Сортировка по любой колонке**. |
| **Чеки и выручка по типу обслуживания** | `/api/dishes/check-distribution`, `/api/revenue/by-channel` | По каждому каналу (Доставка / В зале / С собой) — **доля чеков и доля выручки рядом**. Чеки = **уникальные заказы** (OLAP по `OrderNum`+«Статус»); выручка суммируется из разреза по каналам. Виден перекос: канал может давать мало чеков, но много выручки (или наоборот). | — |
| **Выручка по часам** | `/api/revenue/hourly`, `/api/revenue/hourly-by-channel` | Выручка и чеки по часовым интервалам (11-12 … 22-23). Часы без продаж не схлопываются — ось X равномерна и сопоставима с другими почасовыми графиками. | Режимы **Сумма** / **По статусам** (стек каналов + чипы). Тип графика. |
| **Наполненность чеков по часам** | `/api/dishes/check-fullness` | Распределение чеков по числу позиций (1 / 2 / 3 / 4+) в каждом часе — видно, когда берут одиночные, а когда «дюрюм+суп+напиток». Модификаторы не считаются позициями. Под графиком — **строка-инсайт**: доля одиночных чеков за период и час-пик (потенциал апсейла). | Тумблер **кол-во / доля %**. |
| **Состав чека** | `/api/dishes/check-composition`, `/api/dishes` (провал) | Средняя доля категорий в одном чеке (стек 100%). Доля считается на чек (категория ÷ итог чека), затем усредняется. Модификаторы исключены. **Клик по категории (столбик/легенда) — провал в товары этой категории** (доля товара в категории за период). | Тумблеры **по кол-ву / по выручке** и **за период / по часам**; клик по категории → товары, ✕ закрыть. |
| **Продажи по часам** | `/api/dishes/hourly-breakdown` | **Столбчатая диаграмма по часам со стеком по категориям** (сегменты = категории, высота столбца = выручка/кол-во) + таблица по выбранному часу (блюда/категории, доли с инлайн-барами). Диаграмма всегда в разрезе категорий (стек читаем); таблица — по выбранной группировке. Часы без продаж не схлопываются (ось X совпадает с другими почасовыми). Модификаторы исключены. | Тумблеры **Выручка/Кол-во** и **Объём / Доли** (стек 100%); Категории/Блюда (для таблицы); **клик по столбцу или чипу — выбор часа** (выбранный подсвечен, остальные приглушены); **сортировка по любой колонке**. |
| **Что продаётся по статусам** | `/api/dishes/service-breakdown` | Таблица блюдо/категория × канал (в зале/с собой/доставка): кол-во по каналам, всего и **доля позиции в продажах периода** (а не внутри статуса). Модификаторы исключены. | Блюда/Категории, поиск, **сортировка по любой колонке**. |
| **Меню-инжиниринг** | `/api/dishes` | **Матрица** популярность × маржа на штуку → ⭐ Звёзды / 🐴 Рабочие лошадки / ❓ Загадки / 🐕 Собаки (порог: 70% от средних продаж × средняя маржа), **квадранты залиты фоном**, топ-6 блюд по выручке подписаны прямо на графике; и **ABC-Парето** (A/B/C). ⚠️ Маржа теоретическая (из ТТК) — пока накладные не приходуются. | Матрица/ABC; ABC по выручке/по прибыли. |
| **Продажи блюд** | `/api/dishes` | Таблица: кол-во, выручка, доли (% выручки и кол-ва), **с/с в ₽ и %, маржа %**. С/с — из ТТК (порционная «Итого с/с») по авто-имени или ручной привязке. | Блюда/Категории, поиск + мультивыбор, сортировка по любой колонке. |

> С/с по блюду берётся из тех-карты: порционная «Итого с/с» (продукты+списания+упаковка).
> Если POS-название не совпадает с именем ТТК (напр. «Зурна» ↔ «Искандер дюрюм Zurna»),
> нужна ручная привязка во вкладке **«Блюда ↔ ТТК»**.

## Архитектура и флоу данных

```
                 ┌─────────────┐
   iiko (чтение) │  iikoweb    │  продажи, блюда, (в будущем) остатки
                 └──────┬──────┘
                        │ Playwright логин → cookie-сессия → httpx (JSON)
                        ▼
   Excel ТТК/прайс ─► ┌──────────────────────────┐      ┌──────────────┐
   (импорт openpyxl)  │  Backend (FastAPI)        │◄────►│ SQLite + файлы│
                      │  • синк продаж (cron)     │      │ (volume /data)│
                      │  • импорт ТТК/прайса      │      └──────────────┘
                      │  • API /api/*             │
                      └────────────┬─────────────┘
                                   │ JSON /api/*
                                   ▼
                      ┌──────────────────────────┐
                      │  Frontend (React + Vite)  │  раздаётся nginx,
                      │  дашборд, поставщики, ТТК  │  проксирует /api на backend
                      └──────────────────────────┘
```

**Источник правды по костам — наша БД, в iiko не пишем.** iiko используем только на
чтение. Связь данных: `продажи блюд × ТТК = расход продуктов`; цены из накладных/прайса
дают себестоимость; всё сходится через каталог ингредиентов (номенклатуру).

## Флоу работы (как пользоваться)

1. **Запустить**: `docker compose up -d --build`.
2. **Наполнить данными**: открыть http://localhost:5173 → **Поставщики** →
   «**Импорт из файла**» (или `POST /api/import/ttk-matrix`). Подтянутся поставщики,
   ингредиенты, цены и тех-карты из `backend/seed/ttk_matrix.xlsx`.
3. **Поставщики**: смотреть/редактировать карточки, цены, загружать накладные и прайсы.
4. **Номенклатура ↔ ТТК**: смотреть тех-карты, состав и себестоимость; переходить в
   полуфабрикаты; вести каталог ингредиентов.
5. **Дашборд**: продажи/выручка обновляются из iiko автоматически (синк раз в час +
   ночью; кнопка «Синхронизировать» — вручную).

## Стек

- **Backend**: Python, FastAPI, SQLAlchemy (SQLite), APScheduler (синки),
  Playwright (логин в iiko), httpx, openpyxl (импорт Excel).
- **Frontend**: React 19, Vite, TypeScript, react-router, TanStack Query, recharts.
- **Инфраструктура**: Docker Compose (backend + frontend/nginx), volume для БД и файлов.

---

## Запуск

### Docker (рекомендуется)
```bash
cp .env.example .env      # заполнить IIKO_WEB_PASSWORD
docker compose up -d --build
# frontend: http://localhost:5173   backend: http://localhost:8000
docker compose logs -f backend     # логи
docker compose down                # стоп  (down -v — со сбросом БД)
```

### Локально без Docker
```bash
./start.sh                # ставит зависимости, поднимает backend :8000 и frontend :5173
```

`.env` (см. `.env.example`): `IIKO_WEB_URL`, `IIKO_WEB_LOGIN`, `IIKO_WEB_PASSWORD`,
`IIKO_STORE_ID`, `DATABASE_URL`.

## Структура

```
backend/
  main.py                 точка входа FastAPI, подключение роутеров
  config.py               настройки из .env
  constants.py            коды метрик iiko, типы данных, маппинги
  models.py               таблицы БД
  iiko_web_client.py      клиент iiko (логин Playwright + httpx)
  scheduler.py            синки продаж/блюд (APScheduler)
  storage.py              хранилище загруженных файлов
  weather.py              погода Москвы (Open-Meteo, без ключа)
  utils.py                хелперы: period_range, normalize_phone/email, classify_channel
  importers/ttk_matrix.py импорт Excel «ТТК и матрица продуктов»
  routers/                revenue, dishes, suppliers, nomenclature, imports
  seed/ttk_matrix.xlsx    исходные ТТК/прайс для импорта
frontend/
  src/pages/              Dashboard (вкладки Пульс/Операции/Меню), Suppliers,
                          SupplierCard, NewSupplier, Nomenclature, TtkCard
  src/components/         Sidebar, KpiCards, RevenueChart, WeekdaySummary,
                          HourlyChart, HourlyBreakdown, ChecksDistribution,
                          ServiceBreakdown, CheckComposition, CheckFullness,
                          MenuEngineering, DishTable, DishTtkMapping
  src/api.ts              вызовы API (реэкспорт типов)
  src/types.ts            доменные типы (DTO бэкенда)
  src/constants.ts        пороги/интервалы/палитра/списки (без «магических» чисел)
  src/format.ts           форматтеры чисел + выравнивание часовых осей (fillHourGaps/hourLabel); src/validation.ts — телефон/email
  src/theme.ts            переключение светлой/тёмной темы
  nginx.conf              раздача SPA + прокси /api
```

### Карта API (где что отдаётся)

```
GET  /api/revenue                         выручка по дням (+ summary.prev для KPI-дельт, prev_data для «Погоды»)
GET  /api/revenue/by-weekday              свод по дням недели
GET  /api/revenue/hourly                  выручка/чеки по часам
GET  /api/revenue/by-channel              выручка по дням × канал (OLAP)
GET  /api/revenue/hourly-by-channel       выручка по часам × канал (OLAP)
GET  /api/revenue/kpi-by-channel          KPI доставка/не доставка (брутто)
GET  /api/dishes                          продажи блюд (с/с, маржа; group_by=dish|category)
GET  /api/dishes/check-distribution       чеки по типу обслуживания
GET  /api/dishes/hourly-breakdown         продажи по часам (блюда/категории)
GET  /api/dishes/service-breakdown        блюдо × канал
GET  /api/dishes/check-composition        состав чека (доля категорий)
GET  /api/dishes/check-fullness           наполненность чеков (позиций в чеке)
GET  /api/dishes/order-types              диагностика поля OrderType (пуст у точки)
GET/POST/DELETE /api/dish-mappings        привязка POS-имя ↔ ТТК (для с/с)
GET  /api/suppliers, /api/suppliers/{id}  поставщики; /export → xlsx
POST/PUT/DELETE /api/suppliers/{id}/contacts            контакты/каналы
PUT  /api/suppliers/{id}/products/{price_id}            торговая марка позиции
GET  /api/ingredients, /api/ttk           номенклатура/ТТК (каталог+карточки)
POST /api/import/ttk-matrix               импорт ТТК/прайса;  POST /api/sync — ручной синк
```
Все эндпоинты с периодом принимают `period=day|week|month` **или** `date_from&date_to`.
