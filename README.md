# Iskendy Analytics

Система финансовой аналитики и учёта костов для ресторана «Искенди».
Тянет продажи из iiko, ведёт в собственной БД поставщиков, цены, тех-карты (ТТК) и
себестоимость блюд. Цель — всегда видеть косты и автоматизировать заказ продуктов.

> Точка: **Искенди**, `storeId = 161059`, iiko domain `iskendi`.

---

## Что умеет

- **Дашборд** (продажи из iiko): KPI с дельтами, выручка/часы по каналам обслуживания,
  аналитика состава и наполненности чека, продажи блюд с с/с и маржой, погода.
  Период — Сегодня / Неделя / Месяц **или произвольный диапазон**. Подробно по каждому
  блоку — ниже в разделе [«Дашборд — что показывает каждый блок»](#дашборд--что-показывает-каждый-блок).
- **Поставщики** — список с выгрузкой в Excel; карточка с редактированием реквизитов,
  **несколькими контактами и каналами связи** (телефон/WhatsApp/Telegram/email, с
  валидацией), товарами с ценами и **торговой маркой**, файлами (накладные/прайсы).
- **Номенклатура ↔ ТТК** — каталог ингредиентов, тех-карты с составом и себестоимостью
  (в т.ч. двухуровневые блюдо → полуфабрикат → сырьё) + вкладка **«Блюда ↔ ТТК»**:
  привязка проданных блюд к тех-картам (даёт с/с по блюду, когда POS-имя ≠ имя ТТК).
- **Импорт** из Excel «ТТК и матрица продуктов» — наполняет поставщиков, ингредиенты,
  цены и ТТК одним действием (привязки блюд и торговые марки переживают реимпорт).
- **Светлая/тёмная тема**.
- **Авторизация**: весь дашборд закрыт за экраном входа (логин/пароль). Один общий аккаунт
  из `.env` (`AUTH_USERNAME`/`AUTH_PASSWORD`), сессия — JWT в браузере (срок `JWT_TTL_HOURS`,
  по умолчанию 12 ч), выход — кнопкой «Выйти» в меню. **Обязательно задайте `AUTH_PASSWORD`**
  (пустой пароль = вход невозможен).
- _В разработке_: накладные (косты), остатки из iiko, автозаказ. См. `PLAN.md`.

> Дашборд читает **всё из локальной БД** (SQLite): и выручку по дням, и заказы на уровне
> позиций (`order_items`/`dish_detail`), из которых пересчитываются все разрезы блюд/чеков.
> Поэтому любой период открывается **мгновенно**. Живые запросы к iiko делает только
> фоновый планировщик: ежечасно обновляет последние 7 дней и один раз выкачивает всю
> историю (`backfill`); закрытые дни в iiko неизменны, поэтому тянутся однократно. Для
> диапазонов **старше** сохранённой истории — живой запрос как fallback. Кнопка
> «Синхронизировать» форсирует обновление свежих дней.

## Дашборд — что показывает каждый блок

Период (Сегодня / Неделя / Месяц / произвольный диапазон) задаётся сверху и применяется
ко всем блокам. **Месяц = с 1-го числа текущего месяца по сегодня** (month-to-date), а не
последние 30 дней. **Неделя = с понедельника текущей недели** по сегодня (не последние 7 дней). «Канал» = тип обслуживания: **в зале / с собой / доставка** — берётся из
модификатора категории «Статус» заказа, а доставка — ещё и из меню-категории «Доставка»
(подробности в `IIKO_WEB_API.md`).

Дашборд занимает **всю ширину экрана**; виджеты раскладываются в адаптивную сетку — 2 колонки на широких экранах, 1 на узких (< 1200px). Разбит на **3 вкладки** (KPI-карточки — всегда сверху, над вкладками):
- **Пульс** (деньги и динамика): Выручка по дням · Чеки и выручка по типу обслуживания (в зале / с собой / доставка) · Структура выручки по оплате (Карта / Наличные / Агрегатор).
- **Операции** (когда и как нагружена точка): Свод по дням недели · Ежедневный отчёт по дейпартам · Выручка по часам · Наполненность чеков · Продажи по часам · Выручка по дейпартам. В режиме одного дня почасовой график показывается только на «Пульсе» (на «Операциях» не дублируется).
- **Меню** (что и насколько выгодно): Продажи блюд · Сочетаемость в чеке · Меню-инжиниринг · Состав чека.

Дашборд **без разреза по каналам** — он вынесен на отдельный **дашборд «Доставка»**. Зато сверху есть **галка «С доставкой»** (по умолчанию **выключена** — дашборд открывается без доставки): включи её, чтобы добавить заказы доставки во **все виджеты дашборда** (KPI, «Выручка по дням/часам», «Свод по дням недели», «Наполненность чеков», «Продажи по часам», «Состав чека», «Меню-инжиниринг», «Продажи блюд»). Дельты к прошлому периоду тоже считаются согласованно. **Доставкой считается меню-категория «Доставка» ИЛИ позиция с маркером `_д` в имени** (напр. «Дюрюм_д») — отсев по-позиционно, без правила «Статус». Галка влияет и на «Чеки и выручка по типу обслуживания» (при выкл строка «Доставка» из разреза убирается вовсе). Food cost остаётся по всей точке (с/с по каналам не делится).

**В шапке дашборда** также: кнопка **«Синхронизировать»** (полный синк) с подписью времени последней синхронизации (**сперва время, затем дата** — «14:05 · 23.06», МСК) и **селектор автосинхронизации** «Автосинк: выкл / 5 / 15 / 30 мин / 1 час» — при выбранном интервале дашборд сам делает лёгкий синк (последние 7 дней) и обновляет данные; выбор хранится в localStorage.

Раскладка вкладок задаётся в `frontend/src/pages/Dashboard.tsx` (стейт `tab`).

| **Выручка по дням** | `/api/revenue` | Выручка по дням; **столбцы окрашены по группе дня недели** (Пн / Вт-Ср / Чт / выходные Пт-Вс) с легендой групп — сразу видно тип дня. Чеки = **уникальные заказы** (OLAP по `OrderNum`+«Статус»). Так видно структуру: сколько чеков и денег даёт зал / с собой / доставка. | **Галка «С доставкой»**: выкл → доставка исключается из разреза. |
| **Структура выручки по оплате** | `/api/revenue/by-payment` | Доли выручки и чеков по способу оплаты (**Карта / Наличные / Агрегатор**) за период — две 100%-стопки — плюс **дневной стек-тренд** структуры. Источник — нормализованные оплаты (`order_payments`, сплит-чек разложен, Σ=выручке). | **Галка «С доставкой»**: выкл → оплаты доставочных заказов исключаются (агрегатор = Яндекс Еда = доставка, при выкл исчезает). |
| **Свод по дням недели** | `/api/revenue/by-weekday` | Таблица-агрегат: для каждого дня недели за период — сколько таких дней попало, суммарная и средняя за день выручка, чеки, средний чек. Видно, какие дни недели «вытягивают». Строки **окрашены по группе дня недели** (Пн / Вт-Ср / Чт / выходные), порядок **всегда Пн→Вс**. | **Галка «С доставкой»**. |
| **Выручка по дейпартам** | `/api/revenue/by-daypart` | Столбцы выручки по операционным окнам дня: **Завтрак 08–12 / Ланч 12–16 / Полдник 16–18 / Ужин 18–23 / Ночь** (границы из исторического «свода» ресторана). Над столбцом — доля окна в выручке; под графиком — **таблица** выручка / доля / чеки / средний чек по каждому окну. Видно, когда зарабатывает день. Почасовой источник свёрнут в окна (сумма сходится с `/api/revenue`). Пустые окна (часто Ночь) скрыты. | **Галка «С доставкой»** (выкл → без доставки, вычет по часам). |
| **Ежедневный отчёт по дейпартам** | `/api/revenue/ops-report` | **Плотная таблица-свод** (аналог Excel «Ежедневный ОП»): дни периода по горизонтали (с днём недели), дейпарты по вертикали. Под каждым дейпартом — **Выручка / Средний чек / Гости / Кост %** (food cost = с/с порций по ТТК ÷ выручка, раскрашен по порогам). Справа — Факт за период, Среднее на активный день, доля % дейпарта. Ниже — блок **food cost % по дейпартам × категориям** (Еда / Напитки / Алкоголь). Гости — поле `GuestNum` iiko. Левые колонки липкие, таблица скроллится по горизонтали. **План/Факт/% к плану**: при заданном плане — колонки сравнения (план задаётся дневной нормой на дейпарт × группу дня недели, масштабируется на число дней периода — кнопка «⚙ План» открывает редактор с автозаполнением из истории). Скрыт при одном дне. | **Галка «С доставкой»** (выкл → доставка исключается). |

| **Наполненность чеков по часам** | `/api/dishes/check-fullness` | Распределение чеков по числу позиций (1 / 2 / 3 / 4+) в каждом часе — видно, когда берут одиночные, а когда «дюрюм+суп+напиток». **Позиция = единица товара** (сумма `qty` по заказу: 2 одинаковых кофе = 2 позиции, а не 1), модификаторы не считаются. Под графиком — **строка-инсайт**: доля одиночных чеков за период и час-пик (потенциал апсейла). | Тумблер **кол-во / доля %**. |
| **Состав чека** | `/api/dishes/check-composition`, `/api/dishes` (провал) | Средняя доля категорий в одном чеке (стек 100%). Доля считается на чек (категория ÷ итог чека), затем усредняется. Модификаторы исключены. **Клик по категории (столбик/легенда) — провал в товары этой категории** (доля товара в категории за период). | Тумблеры **по кол-ву / по выручке** и **за период / по часам**; клик по категории → товары, ✕ закрыть. |
| **Продажи по часам** | `/api/dishes/hourly-breakdown` | **Столбчатая диаграмма по часам со стеком по категориям** (сегменты = категории, высота столбца = выручка/кол-во) + таблица по выбранному часу (блюда/категории, доли с инлайн-барами). Диаграмма всегда в разрезе категорий (стек читаем); таблица — по выбранной группировке. **В режиме «Категории» строки таблицы разворачиваются в блюда** этой категории за выбранный час (drill-down ▶/▼) — мелкие позиции становятся видны. Часы без продаж не схлопываются (ось X совпадает с другими почасовыми). Модификаторы исключены. | Тумблеры **Выручка/Кол-во** и **Объём / Доли** (стек 100%); Категории/Блюда (для таблицы); **клик по столбцу или чипу — выбор часа**; **разворот категории в блюда**; **сортировка по любой колонке**. |
| **Сочетаемость в чеке** | `/api/dishes/basket` | Market basket: что чаще берут вместе в одном чеке. **Крупная тепловая матрица** топ-позиций (ячейка = в скольких чеках встречаются обе). Всегда по блюдам. **Порядок осей** — тумблер «По популярности / А–Я» (фиксирован, при клике не двигается). **Клик по подписи блюда** (строки/столбца) — **подсветка**: само блюдо и всё, что с ним берут в чеках, явно выделяются (акцент + рамка), остальное гасится (повторный клик / «сбросить» — отмена). | Галка «С доставкой». |
| **Меню-инжиниринг** | `/api/dishes` | **Матрица** популярность × маржа на штуку → ⭐ Звёзды / 🐴 Рабочие лошадки / ❓ Загадки / 🐕 Собаки (порог: 70% от средних продаж × средняя маржа), **квадранты залиты фоном**, топ-6 блюд по выручке подписаны прямо на графике; и **ABC-Парето** (A/B/C) + **таблица блюд** под диаграммой (ранг, класс, выручка/прибыль, накопит. %, **кост %** раскрашен по порогам food cost). Блюда без ТТК-привязки (нет с/с) в матрицу и в ABC-по-прибыли **не попадают** (иначе считались бы как 100% маржа); их число подписано под матрицей, а в таблице кост % у них «—». ⚠️ Маржа теоретическая (из ТТК) — пока накладные не приходуются. | Матрица/ABC; ABC по выручке/по прибыли. |
| **Продажи блюд** | `/api/dishes` | Таблица (колонки слева направо): Блюдо / Группа / Кол-во / **Выручка** / Доля кол-ва / Доля продаж / **План с/с / Факт с/с / План кост % / Факт кост % / Δ кост / Маржа %** (абсолютные числа идут перед долями). «Кост» = с/с ÷ цена реализации. План с/с — из ТТК (порционная «Итого с/с») по авто-имени или ручной привязке. **Факт с/с / кост / Δ — пока заглушка «—»** (появятся после учёта накладных). Без привязки ТТК — «—» (такие строки сортируются вниз). | Блюда/Категории; в режиме «Блюда» единый поиск-мультивыбор подсказывает **и категории, и товары** — можно набрать сразу несколько категорий (чип с меткой «кат.») и отдельные товары (подсказки товаров сужаются выбранными категориями); таблица = блюда выбранных категорий ∪ выбранные товары. Сортировка по любой колонке. |

> С/с по блюду берётся из тех-карты: порционная «Итого с/с» (продукты+списания+упаковка).
> Если POS-название не совпадает с именем ТТК (напр. «Зурна» ↔ «Искандер дюрюм Zurna»),
> нужна ручная привязка во вкладке **«Блюда ↔ ТТК»**.

### Дашборд доставки (`/delivery`, пункт «🛵 Доставка» в сайдбаре)

Доставка вынесена в отдельный раздел, чтобы не смешивать метрики: общие KPI считаются по
`REV_GROSS`, а каналы — по OLAP, и в одной карточке они не сходились бы. Внутри дашборда
доставки **всё из OLAP** (`/api/revenue/kpi-by-channel` + `/api/revenue/by-channel`), поэтому
цифры согласованы между собой.

| Блок | Источник (API) | Что показывает |
|---|---|---|
| **KPI доставки** | `/api/revenue/kpi-by-channel` | Выручка доставки · Чеков · Средний чек · **Доля доставки** (% выручки и % чеков от всего). |
| **Выручка по дням** | `/api/revenue/by-channel` | Стек по дням: доставка и остальное (зал+с собой) — виден вклад доставки в каждый день. |

Свой выбор периода (Сегодня / Неделя / Месяц / диапазон), как на главном дашборде.

## Архитектура и флоу данных

```
                 ┌─────────────┐
   iiko (чтение) │  iikoweb    │  продажи, блюда, (в будущем) остатки
                 └──────┬──────┘
                        │ Playwright логин → cookie-сессия → httpx (JSON)
                        ▼
   Excel ТТК/прайс ─► ┌──────────────────────────┐      ┌──────────────┐
   (импорт openpyxl)  │  Backend (FastAPI)        │◄────►│ SQLite + файлы│
                      │  • синк продаж (cron)     │      │ (volume /data)│
                      │  • импорт ТТК/прайса      │      └──────────────┘
                      │  • API /api/*             │
                      └────────────┬─────────────┘
                                   │ JSON /api/*
                                   ▼
                      ┌──────────────────────────┐
                      │  Frontend (React + Vite)  │  раздаётся nginx,
                      │  дашборд, поставщики, ТТК  │  проксирует /api на backend
                      └──────────────────────────┘
```

**Источник правды по костам — наша БД, в iiko не пишем.** iiko используем только на
чтение. Связь данных: `продажи блюд × ТТК = расход продуктов`; цены из накладных/прайса
дают себестоимость; всё сходится через каталог ингредиентов (номенклатуру).

---

## Флоу работы (как пользоваться)

1. **Запустить**: `docker compose up -d --build`.
2. **Наполнить данными**: открыть http://localhost:5173 → **Поставщики** →
   «**Импорт из файла**» (или `POST /api/import/ttk-matrix`). Подтянутся поставщики,
   ингредиенты, цены и тех-карты из `backend/seed/ttk_matrix.xlsx`.
3. **Поставщики**: смотреть/редактировать карточки, цены, загружать накладные и прайсы.
4. **Номенклатура ↔ ТТК**: смотреть тех-карты, состав и себестоимость; переходить в
   полуфабрикаты; вести каталог ингредиентов.
5. **Дашборд**: продажи/выручка обновляются из iiko автоматически (серверный синк раз в
   час + ночью; кнопка «Синхронизировать» — вручную; селектор «Автосинк» в шапке —
   клиентская автосинхронизация раз в 5/15/30 мин или час).

## Запуск

### Docker (рекомендуется)
```bash
cp .env.example .env      # заполнить IIKO_WEB_PASSWORD и AUTH_PASSWORD
docker compose up -d --build
# frontend: http://localhost:5173   backend: http://localhost:8000
docker compose logs -f backend     # логи
docker compose down                # стоп  (down -v — со сбросом БД)
```

### Локально без Docker
```bash
./start.sh                # ставит зависимости, поднимает backend :8000 и frontend :5173
```

`.env` (см. `.env.example`): `IIKO_WEB_URL`, `IIKO_WEB_LOGIN`, `IIKO_WEB_PASSWORD`,
`IIKO_STORE_ID`, `DATABASE_URL`, `TIMEZONE` (часовой пояс ресторана, по умолчанию
`Europe/Moscow` — задаёт границы «сегодня/неделя/месяц» и расписание синков),
`CACHE_TTL_SECONDS` (TTL in-memory кэша живых чтений, дефолт 60 c; 0 — выключить),
**`AUTH_USERNAME`/`AUTH_PASSWORD`** (вход в дашборд — обязателен непустой пароль),
`JWT_SECRET` (секрет подписи токена; при пустом выводится из пароля), `JWT_TTL_HOURS`
(срок сессии, дефолт 12 ч).

### Карта API (где что отдаётся)

```
GET  /api/revenue                         выручка по дням (+ summary.prev для KPI-дельт, prev_data для «Погоды»); ?include_delivery=false — без доставки
GET  /api/revenue/by-weekday              свод по дням недели (?include_delivery=false — без доставки)
GET  /api/revenue/by-daypart              выручка/чеки/ср.чек по дейпартам Завтрак/Ланч/Полдник/Ужин/Ночь (?include_delivery=false — без доставки)
GET  /api/revenue/hourly                  выручка/чеки по часам (?include_delivery=false — без доставки)
GET  /api/revenue/ops-report              ежедневный операционный отчёт: дни × дейпарты (выручка/чеки/гости/ср.чек/food cost %/план; ?include_delivery=false — без доставки)
GET  /api/plan                            план: матрица норм (дейпарт × группа дня недели)
PUT  /api/plan                            сохранить план (нормы на день)
POST /api/plan/seed-from-history?months=N автозаполнить план средними дневными нормами из истории
GET  /api/revenue/by-payment              структура выручки по оплате (Карта/Наличные/Агрегатор): доли + дневной тренд (?include_delivery=false)
GET  /api/revenue/by-channel              выручка по дням × канал (OLAP)
GET  /api/revenue/hourly-by-channel       выручка по часам × канал (OLAP)
GET  /api/revenue/kpi-by-channel          KPI доставка/не доставка (брутто)
GET  /api/dishes                          продажи блюд (с/с, маржа; group_by=dish|category); ?include_delivery=false — без доставки
GET  /api/dishes/check-distribution       чеки по типу обслуживания; ?include_delivery=false — без доставки
GET  /api/dishes/hourly-breakdown         продажи по часам (блюда/категории; item несёт category для drill); ?include_delivery=false
GET  /api/dishes/basket                   матрица сочетаемости (что берут вместе; group=category|dish); ?include_delivery=false
GET  /api/dishes/service-breakdown        блюдо × канал
GET  /api/dishes/check-composition        состав чека (доля категорий); ?include_delivery=false — без доставки
GET  /api/dishes/check-fullness           наполненность чеков (позиций в чеке); ?include_delivery=false — без доставки
GET  /api/dishes/order-types              диагностика поля OrderType (пуст у точки)
GET/POST/DELETE /api/dish-mappings        привязка POS-имя ↔ ТТК (для с/с)
GET  /api/suppliers, /api/suppliers/{id}  поставщики; /export → xlsx; DELETE /api/suppliers/{id} — удаление (каскад)
POST/PUT/DELETE /api/suppliers/{id}/contacts            контакты/каналы
PUT  /api/suppliers/{id}/products/{price_id}            торговая марка позиции
GET  /api/ingredients, /api/ttk           номенклатура/ТТК (каталог+карточки)
POST /api/import/ttk-matrix               импорт ТТК/прайса
POST /api/sync                            синк: без параметра — полный (31 день), ?days=N — лёгкий за N дней (автосинк); сбрасывает кэш
GET  /api/sync/last                       время последней успешной синхронизации
```
Все эндпоинты с периодом принимают `period=day|week|month` **или** `date_from&date_to`.
