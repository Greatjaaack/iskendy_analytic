# Журнал работ — Iskendy Analytics

История: что хотели, что сделано, что осталось. Обновляется по ходу проекта.
Связанные документы: `PLAN.md` (общая дорожная карта), `IIKO_WEB_API.md` (API iiko),
`~/.claude/plans/typed-baking-frost.md` (детальный план Фаз 1–3).

---

## Цель проекта

Превратить дашборд аналитики ресторана «Искенди» в систему учёта костов:
поставщики, ТТК, накладные, автозаказ. **В iiko не пишем** — источник правды по
костам/ТТК/поставщикам — собственная БД; iiko только читаем (продажи, в будущем остатки).

## Ключевые решения

- Запись в iiko — **нет**, ведём всё у себя.
- Остатки для автозаказа — из склада iiko (склад пока не ведётся → временно расчётные).
- Накладные — Excel + PDF-сканы (распознавание сканов: решим перед Фазой 4).
- Стартовое наполнение — импорт файла «ТТК и матрица продуктов.xlsx».
- Хранилище файлов — локально в Docker volume (`/data/files`).

## Сделано

### Инфраструктура
- **Docker**: весь стек в контейнерах (`docker compose up -d --build`).
  Backend `:8000`, frontend `:5173` (nginx раздаёт SPA + проксирует `/api`).
  БД (SQLite) и файлы — в volume `backend-data`.
- Починены преexisting TS-ошибки сборки фронта.

### Фаза 0 — логин в iiko (готово)
- **Причина падения**: код заполнял форму через `fill()`, который не триггерит
  валидаторы Angular → кнопка `SIGN IN` оставалась `disabled` → `click()` падал по
  таймауту. Плюс селекторы были под русский UI («Войти»), а интерфейс английский.
- **Фикс** (`backend/iiko_web_client.py`): `domcontentloaded`, поля по `name`,
  ввод через `type()` с задержкой, сабмит по Enter. Проверено живым входом —
  продажи/блюда тянутся.

### Фазы 1–3 — Фундамент, Поставщики, Номенклатура ↔ ТТК (готово)
- **Модель БД** (`backend/models.py`): `Supplier`, `SupplierFile`, `Ingredient`,
  `SupplierPrice`, `Ttk`, `TtkIngredient`. Хранилище файлов — `backend/storage.py`.
- **Импорт xlsx** (`backend/importers/ttk_matrix.py`): разбор 68 листов —
  «Матрица поставщиков» (прайс/фасовки/даты), «Матрица продуктов», «Сводная»
  (категории), лист-на-ТТК. Двухуровневые карты (блюдо → п/ф → сырьё) резолвятся
  по нормализованному имени. Себестоимость берётся готовой из файла. Идемпотентен.
- **API** (`backend/routers/`): `imports.py` (`POST /api/import/ttk-matrix`),
  `suppliers.py` (CRUD + upload/download файлов), `nomenclature.py`
  (ингредиенты, ТТК, карточки).
- **Frontend**: роутинг (`react-router-dom`) + сайдбар; страницы Поставщики,
  Карточка поставщика, Новый поставщик, Номенклатура↔ТТК (вкладки), Карточка ТТК.

### Доработки дашборда (UX)
- **Выручка по дням**: выбор типа графика (Область/Линия/Ступени/Столбцы), подписи
  дней недели, фильтр по дням недели (Все/Будни/Выходные/конкретный день).
- **Продажи по часам**: новый блок, интервалы 11-12 … 22-23 (`HourlyChart`,
  эндпоинт `GET /api/revenue/hourly`).
- **Продажи блюд**: группировка Блюда/Категории, доли (% выручки и кол-ва),
  сортировка по колонкам, поиск-фильтр.
- **Поставщики**: список-таблица вместо плитки.
- **Кодстайл**: бизнес-пороги/интервалы/палитра вынесены в `frontend/src/constants.ts`
  с комментариями.

> Важно: `GET /api/dishes` и `GET /api/revenue/hourly` теперь **живой запрос в iiko**
> за выбранный период (раньше блюда читались из БД-агрегата по окну синка — из-за чего
> «неделя» показывала «Нет данных»). Выручка по дням осталась из БД.

### Доработки дашборда — волна 2
- **Произвольный период** (календарь): все эндпоинты принимают `date_from/date_to`.
- **Распределение чеков по типу обслуживания** (Доставка/В зале/С собой) —
  `GET /api/dishes/check-distribution`, компонент `ChecksDistribution`. Источник —
  количество модификаторов категории «Статус».
- **Чеки в графике «Выручка по дням»** — отдельная правая ось.
- **«Продажи по часам»** — переключатель типа графика (как у выручки) + выручка/чеки.
- **«Статус» убран из «Продаж блюд»** — это модификаторы (`productType=MODIFIER`),
  не блюда. Фильтр в `GET /api/dishes`.
- **С/С %** в «Продажах блюд** (= себестоимость/выручка; источник себестоимости —
  метрика iiko `PRODUCTS_USAGE_THEO_AMT`).
- **Погода в Москве** по дням (`backend/weather.py`, Open-Meteo, без ключа) — строка
  под графиком выручки + шапка «Сегодня» (дата, день недели, погода).
- **Светлая/тёмная тема** — переключатель в сайдбаре; палитра на CSS-переменных
  (`index.css` + `theme.ts`), цвета берутся через `var(--…)`/`COLORS`.
- **Что берут по часам** (#3): разбивка продаж по блюдам/категориям за каждый часовой
  интервал — через **OLAP-движок iiko**. `GET /api/dishes/hourly-breakdown`,
  компонент `HourlyBreakdown`.
- **Привязка блюдо ↔ ТТК** (вкладка в «Номенклатуре», `DishTtkMapping`): POS-имена
  продаж iiko не совпадают с именами ТТК → сопоставляем вручную; по привязке считается
  с/с % по блюду (`DishMapping`, `GET/POST/DELETE /api/dish-mappings`).
- **Кодстайл**: списки/коды метрик/типы данных/OLAP-поля/маппинги вынесены в
  `backend/constants.py` и `frontend/src/constants.ts`; добавлены докстринги.
- **Приведение к код-стайлу** (`code-style.md`): backend прогнан `black`+`isort` (line-length
  100), `flake8` — 0 нарушений; конфиг через pydantic `Settings` (`config.py`) вместо
  `os.getenv`; `HTTPException` со `status.HTTP_*`; `logger.exception` с полным traceback;
  тулинг в `backend/pyproject.toml` + `.flake8`.

> OLAP отреверсен по JS-бандлу фронта iikoweb. Поток (в `iiko_web_client.olap_sales`):
> `POST /api/olap/init` (тело=запрос) -> `GET /api/olap/fetch-status/{hash}` (поллинг до
> SUCCESS) -> `POST /api/olap/fetch/{hash}/simple` (тело=запрос) -> `result.rows`.

### Исправленные по ходу баги
- TS-импорты типов (сборка фронта).
- Импортёр: приведение имён к строке; канон-имя ТТК по имени листа (B1 местами с

### С/с блюд — корректная порционная (#2) + доставка `_д` (#3)
- **Баг**: с/с считалась неверно почти везде. Причины: (1) «Сводная» (источник
  `cost_full`) заполнена лишь у ~11/63 ТТК; (2) fallback на `cost_total` брал с/с за
  ВЕСЬ выход карты (батч) — напр. чай 28.84 за 4800 мл вместо ~8 за чашку → завышение
  в разы; (3) имена POS↔лист↔«Сводная» не совпадают (POS «Зурна» / лист «Искандер
  дюрюм Zurna» / «Сводная» «Iskendy Durum Zurna»).
- **Фикс**: `ttk_matrix._portion_cost()` парсит строку «Итого с/с» (с/с блюда +
  списания + упаковка) с каждого листа ТТК → это и есть порционная с/с. Стала
  основным источником `cost_full`, «Сводная» — запасной. Покрытие порционной с/с:
  **27/34** прямых блюд (Зурна 172.21, Дюрюм 116.56, Чай 8.25, Кебаб дюрюм 209.87…);
  без неё — 6 комбо (нужен расчёт из компонентов) и 1 кофе со сломанной формулой
  `#N/A` в самом файле.
- В `dishes.py` убран опасный fallback на `cost_total` (`_ttk_portion_cost` берёт
  только `cost_full`); постфикс `_д` (доставка) срезается перед матчингом с/с
  (`utils.split_delivery`), а в ответ добавлено поле `channel` («доставка»/"").
- Фронт: бейдж «доставка» у `_д`-позиций в «Продажах блюд».
- ⚠️ Нужно заново прогнать импорт (`POST /api/import/ttk-matrix`), чтобы пересчитать
  `cost_full` новым парсером.

### Телефоны/контакты поставщика (#1) — готово
- Таблица `supplier_contacts` (телефон + контактное лицо + заметка, много на
  поставщика); у `Supplier` убраны одиночные `phone`/`contact_person`.
- Валидация номера: `utils.normalize_phone()` (бэк, pydantic `field_validator`) +
  `validation.normalizePhone()` (фронт) → нормализуют в `+7XXXXXXXXXX`.
- Эндпоинты `POST/PUT/DELETE /api/suppliers/{id}/contacts[/{cid}]`; контакты в выдаче
  поставщика. UI: редактор телефонов в форме создания + секция «Контакты» в карточке

- ⚠️ Схема БД изменилась → пересоздать БД (`docker compose down -v`).

### Разрез блюдо × канал обслуживания (#4) — готово, проверено вживую
- Выяснено на живом OLAP: `OrderType`/`OrderServiceType` у точки почти пусты — тип
  обслуживания задаётся **модификатором категории «Статус» на уровне заказа**.
- `GET /api/dishes/service-breakdown` (group=dish|category): OLAP SALES группирует
  `[OrderNum, DishCategory, DishName]`; у каждого заказа из строки «Статус» берётся
  канал и относится ко всем блюдам заказа; постфикс `_д` форсит доставку.
- Диагностика: `GET /api/dishes/order-types` (показал, что OrderType пуст).
- UI: компонент `ServiceBreakdown` (таблица блюдо×каналы+всего, поиск, dish/category).
- Проверено: Дюрюм за месяц — зал 163 / с собой 126 / доставка 87.

### Канал доставки — по категории «Доставка», а не по `_д` (уточнение от владельца)
- Бизнес-правило: всё, что в меню-категории **«Доставка»**, продано в доставку.
  Канал теперь: категория «Доставка» → доставка; иначе — «Статус» заказа. Постфикс
  `_д` для определения канала и бейджа БОЛЬШЕ НЕ используется.
- `_д`-позиции (Дюрюм_д, Фри_д, Айран_д) лежат НЕ в категории «Доставка», а в своих
  (Дюрюмы/Снэки/Напитки) — их канал даёт «Статус» заказа (на практике = доставка).
- `_д` оставлен ТОЛЬКО невидимо в `get_dishes` — срез постфикса для подстановки с/с
  (Дюрюм_д берёт с/с «Дюрюм»: 116.6), иначе пришлось бы привязывать вручную.
- Проверено вживую: категория «Доставка» = 321 шт, все в доставку (зал 0 / с собой 0).

### С/с: универсальный разбор листов + убран `_д` из с/с
- Парсер ТТК сделан НЕзависимым от раскладки колонок (`_col_map`: ищет «с/с руб.» и
  «Выход» по шапке). Комбо-листы имеют сдвинутую раскладку (с/с в col5, без колонок
  потерь) — теперь их с/с парсится: КОМБО ЛАЙТ 109.73, ОБЕД 228.67, ЛАЙТ-2 147.98,
  ОБЕД-2 266.92. Цепочка cost_full: «Итого с/с» → «Сводная» → «Итого» рецепта (резерв
  для комбо). Итог: **cost_full у 57/63 ТТК** (не парсятся «Комбо-1_1/_2» — в выгрузке
  `#REF!`, нужна живая Google-таблица).
- Постфикс `_д` УБРАН из подстановки с/с (по решению владельца): имя продажи матчится
  как есть. Доставочные позиции (`_д`, комбо, «Зурна» и пр.) costятся через привязку.
- Канал доставки — только по категории «Доставка» (+ «Статус» заказа), `_д` нигде.

### Привязки блюдо↔ТТК: засеяны + 2 бага исправлены
- Засеяно 11 привязок POS→ТТК (Зурна→Искандер дюрюм Zurna, Дюрюм_д→Дюрюм, Нарсоган(+
  доставка)→Дюрюм Нарсоган, Мини(+дюрюм мини доставка)→Дюрюм mini, Балык→Балык Дюрюм,
  Фри_д→Фри на продажу, Айран_д→Айран, Чечевичный суп_д→Чечевичный суп, Халва→Манная халва).
- **Баг (коллизия имён п/ф↔блюдо)**: «Чечевичный суп» есть и блюдом (порция 36.94), и
  п/ф-котлом (батч 231.81 из «пф Чечевичный суп»). Авто-матчинг хватал батч. Фикс:
  в `_dish_unit_cost` авто-матч только по НЕ-п/ф (`is_semi=False`).
- **Баг (привязки ломались при реимпорте)**: хранятся по ttk_id, а ТТК пересоздаются с
  новыми id. Фикс: `import_ttk_matrix` запоминает имя цели и перелинковывает привязки
  по `name_norm` после импорта (предпочитая не-п/ф). Проверено: после реимпорта 11/11 целы.

## Осталось (дорожная карта)

- **Привязки для комбо и «Напиток»/«Кола»** (данные): отложены по решению владельца.
  Комбо в seed-файле частично сломаны (`#REF!`) — возможно, нужна живая Google-таблица.
- **Заполнить привязки блюдо↔ТТК** (данные, не код): владельцу сопоставить блюда во
  вкладке «Блюда ↔ ТТК» — тогда с/с % покроет все позиции (механизм готов).
- **Фаза 4 — Накладные**: загрузка Excel/PDF, парсинг, привязка позиций к
  номенклатуре, учёт закупочных костов. Перед стартом — выбрать распознаватель
  сканов (Claude API vs Tesseract).
- **Фаза 5 — Остатки из iiko**: реверс эндпоинта остатков склада (когда склад в iiko
  начнут вести), синк в `stock_levels`.
- **Фаза 6 — Автозаказ**: расход = продажи × ТТК, сверка с остатками, ночной cron,
  формирование и экспорт заявок по поставщикам.
