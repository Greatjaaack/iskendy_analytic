# План развития Iskendy Analytics

Документ для согласования объёма работ перед реализацией. Описывает: что меняется,
архитектуру, модель данных, страницы и порядок сборки по фазам.

## Зафиксированные решения

| Вопрос | Решение |
|---|---|
| Запись в iiko | **НЕ пишем.** Косты, ТТК, поставщиков, заказы ведём в собственной БД. iiko только на чтение. |
| Остатки для автозаказа | **Из склада iiko** (новый read-эндпоинт остатков). |
| Форматы накладных | **Excel (.xlsx/.xls)** + **PDF-сканы** (не фото). Сканы → через распознавание. |
| Порядок сборки | Сначала фундамент-справочники, затем зависимые модули (см. Фазы). |

> Важная корректировка исходной задачи: пункты «заносишь в Айку веб» и «ТТК в Айке
> внести» меняются на «ведём у себя». Всё, что раньше шло «в iiko», теперь хранится
> в нашей системе как источник правды по костам.

## Предусловие (Фаза 0) — починить логин в iiko

Сейчас Playwright-логин падает (`Page.click: Timeout` на кнопке «Войти») — данные не
тянутся вообще. Без рабочего логина не будет ни продаж, ни остатков. Чиним первым делом:
поправить селекторы формы по реальной вёрстке, прогнать в видимом режиме до успеха.

## Архитектура и стек (дополнения к текущему)

- **Backend (FastAPI)** — добавляем модули: загрузка/парсинг файлов, поставщики, ТТК,
  заявки. Хранилище файлов — на диске в volume (`/data/files`), путь пишем в БД.
- **Парсинг накладных:**
  - Excel → `openpyxl`/`pandas` (надёжно, правила по шаблону поставщика).
  - PDF-скан → распознавание. Рекомендую vision-модель (Claude API) — даёт структурный
    JSON по таблице накладной с высокой точностью. Альтернатива — локальный Tesseract
    (бесплатно, но точность ниже, требует больше ручной правки). ← нужно подтвердить.
- **БД** — пока SQLite (как сейчас). При росте объёма легко перенести на Postgres.
- **Frontend (React)** — появляется несколько страниц → добавляем роутинг
  (`react-router-dom`) и боковое меню. Сейчас одностраничный дашборд.
- **iiko (чтение):** продажи (есть) + **остатки склада** (новый эндпоинт, реверс).

## Модель данных (новые таблицы)

```
suppliers              поставщики: name, contact_person, phone, address,
                       min_delivery, comment
supplier_files         файлы поставщика: supplier_id, path, type(накладная/прайс), uploaded_at
products               зеркало номенклатуры iiko: iiko_id(uuid), name, category, unit
supplier_products      товар у поставщика: supplier_id, supplier_item_name,
                       product_id(=привязка к номенклатуре, может быть пустой),
                       pack/фасовка, current_price
price_history          история цен: supplier_product_id, price, date, invoice_id
invoices               накладные: supplier_id, number, date, total, file_id, status
invoice_lines          строки накладной: invoice_id, supplier_item_name,
                       matched_product_id, qty, unit, price, sum
ttk                    тех-тех карты: name, dish_id(iiko), выход
ttk_ingredients        состав ТТК: ttk_id, product_id, норма закладки, unit
stock_levels           остатки из iiko: product_id, qty, unit, snapshot_date
orders                 заявки: supplier_id, date_for, status, total
order_lines            строки заявки: order_id, product_id, qty, unit, price, sum
```

**Связующая логика (расчёт расхода → заказа):**
`продажи блюд (из iiko) × ttk_ingredients` = расход продуктов →
сверка с `stock_levels` → потребность → группировка по поставщикам = `orders`.

## Страницы (детально)

### Дашборд (есть)
Выручка, блюда, KPI. Без изменений (позже добавим виджет костов).

### 1. Накладные (приёмки и косты)
- Загрузка файла (Excel/PDF-скан), выбор поставщика.
- Распознавание: шапка (дата, №, поставщик, сумма) + позиции (наименование, кол-во,
  ед.изм, цена, сумма).
- Автопривязка позиций к номенклатуре iiko по сохранённым соответствиям (стр. 5);
  непривязанные правим вручную.
- Экран предпросмотра → подтверждение пользователем.
- Сохранение: строки костов + история цен; оригинал файла → в карточку поставщика.
- **Результат:** актуальные закупочные цены, косты всегда видны.

### 2. Поставщики (справочник)
- Список/карточки: название, контакт, телефон, мин.поставка.
- Карточка поставщика:
  - Реквизиты: название, контактное лицо, телефон, адрес, мин.поставка, комментарий.
  - Товары поставщика + цена по каждому (текущая + история, график).
  - История заказов и накладных с прикреплёнными файлами (Excel/PDF).
- **Результат:** один экран — всё о поставщике.

### 3. Новый / редактирование поставщика
- Форма реквизитов.
- Загрузка файлов (прайс/накладные) → авто-парсинг предзаполняет товары и цены.
- **Результат:** завёл контрагента + закинул прайс → товары подтянулись сами.

### 4. Автозаказ (ночное формирование заявок)
- Ночной cron: продажи × ТТК → расход → сверка с остатками iiko → потребность.
- Группировка заявок по поставщикам (позиции, кол-во, цена из последней накладной).
- Экран заявок: правка количеств, учёт мин.поставки.
- Экспорт/отправка поставщику (Excel/PDF).
- **Результат:** утром готовы заявки на день вперёд по реальному расходу.

### 5. Номенклатура ↔ ТТК
- Список номенклатуры iiko + справочник ТТК.
- Ведение ТТК (состав: ингредиент + норма).
- Привязки: наименование поставщика → позиция iiko → ТТК, где используется.
- **Результат:** единый словарь, связывающий накладные, расход и косты.

## Фазы реализации (порядок)

- **Фаза 0 — Починка логина iiko** (предусловие, без него ничего не тянется).
- **Фаза 1 — Каркас:** роутинг + меню на фронте; базовые таблицы; хранилище файлов.
- **Фаза 2 — Поставщики (стр. 2, 3):** справочник, карточки, загрузка файлов.
- **Фаза 3 — Номенклатура ↔ ТТК (стр. 5):** импорт номенклатуры, ведение ТТК, привязки.
- **Фаза 4 — Накладные (стр. 1):** парсинг Excel → затем PDF-сканы; косты, история цен.
- **Фаза 5 — Остатки iiko:** реверс эндпоинта остатков, синк в `stock_levels`.
- **Фаза 6 — Автозаказ (стр. 4):** расчёт расхода, ночной cron, экран и экспорт заявок.

## Открытые вопросы (нужно для старта)

1. **Склад в iiko:** реально ли в точке ведётся складской учёт и есть остатки?
   (Если нет — автозаказ строить не на чем, перейдём на расчётные остатки.)
2. **ТТК:** в каком виде дашь карты (Excel/PDF) — чтобы спланировать импорт.
3. **Распознавание сканов:** ок ли использовать Claude API (vision, платно, нужен ключ)
   для PDF-сканов, или ставим бесплатный Tesseract (точность ниже)?
4. **Фасовки/единицы:** как считать пересчёт (кг ↔ шт ↔ коробка)? Нужен справочник
   фасовок у каждого поставщика, иначе кол-во в заявке будет неточным.
5. **Хранилище файлов:** локально в volume контейнера — устраивает? (или нужно облако).
