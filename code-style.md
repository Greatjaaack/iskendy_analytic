# Code style

Свод стилевых правил проекта. Backend (Python) приведён к этому стилю (`black`, `isort`,
`flake8`, `mypy` — конфиг в `backend/pyproject.toml` и `backend/.flake8`). Frontend (TS/React)
живёт по своим конвенциям — см. раздел 13.

---

## 1. Форматирование (Python)

- **Отступы — 4 пробела**, не табы.
- **Длина строки — до 100 символов.**
- **Кавычки — двойные** (`"текст"`); одинарные — если внутри есть двойные.
- **Пустые строки:** одна между методами класса, две между функциями/классами верхнего уровня.
- **Пробелы вокруг операторов** (`x = 1`, `a + b`), без пробела перед `,`/`:`, один после.
- **В конце файла — одна пустая строка**, без хвостовых пробелов.

Инструменты (через pre-commit): `black` (line-length 100), `isort` (profile black),
`mypy`, `flake8` (max-line-length 100, ignore `E203`/`W503`). Споры о формате решает тула.

## 2. Именование

| Сущность | Стиль | Пример |
|----------|-------|--------|
| переменные, функции | `snake_case` | `appeal_text`, `score_request()` |
| классы | `PascalCase` | `AppealClassifier` |
| константы | `UPPER_SNAKE` | `MAX_TOKENS` |
| приватное | префикс `_` | `_build_prompt()` |

Имя говорит **что это**, без сокращений (в меру). TS: `camelCase` для значений, `PascalCase`
для компонентов/типов, `UPPER_SNAKE` для констант.

## 3. Аннотации типов

Все публичные функции/методы аннотируем; внутри тела — по необходимости. Для сложных
структур — `dataclass`/`TypedDict` (Python), `interface`/`type` (TS).

## 4. Маленькие функции, ранний возврат

Не помещается на экран — разбиваем. Вместо вложенных `if` — guard clauses (ранний `return`).

## 5. Stepdown («газетная» структура)

Вызывающая функция — выше вызываемой; общее сверху, детали ниже. Точка входа
(`main`/`if __name__ == "__main__"`) — внизу файла. В классе: публичные методы сверху,
приватные хелперы под ними.

## 6. Без «магических» значений

Значение со смыслом — в константу/конфиг. **Секреты и окружение — через `BaseSettings`**
(один типизированный класс на приложение), а не `os.environ` по коду.
В проекте: `backend/config.py` (`Settings`), `backend/constants.py` (коды/маппинги),
`frontend/src/constants.ts` (пороги/палитра/списки).

## 7. Запятые и переносы

Хвостовая запятая после последнего элемента; при переносе — каждый элемент на своей строке.
Чинит `black` автоматически.

## 8. Промпты/большие шаблоны — в отдельные файлы

Длинные текстовые шаблоны выносим в `.txt`/`.yaml`, не хардкодим в коде.
(В проекте промптов нет; для seed-данных используем `backend/seed/`.)

## 9. Логирование

`logging`, не `print`. Уровни по смыслу. `logger.info("...", extra={...})` для структурных полей.

## 10. Ошибки — явно и со всем телом

Не глотать исключения. Ловим конкретные типы, логируем **полный traceback**
(`logger.exception(...)` внутри `except`), при необходимости пробрасываем.
**Веб-ошибки — `HTTPException` со `status.HTTP_*`**, без захардкоженных кодов.

```python
raise HTTPException(status.HTTP_404_NOT_FOUND, "поставщик не найден")
```

## 11. Импорты

Группы: стандартная библиотека → сторонние → локальные, разделены пустой строкой
(чинит `isort`). Без `from module import *`.

## 12. Комментарии объясняют «почему», а не «что»

Коммент — там, где неочевидное решение или внешний контекст, а не пересказ кода.

---

## 13. Frontend (TS/React)

- Формат — Prettier/ESLint (2 пробела — конвенция экосистемы; Python-правило 4 пробелов сюда
  не переносим). Линт: `npm run lint`.
- **Тип-only импорты:** `import type { Foo } from "..."` для типов (требует `verbatimModuleSyntax`).
- Бизнес-пороги/интервалы/палитра/списки — в `src/constants.ts`, не в разметке.
- Цвета — через CSS-переменные (`var(--…)`)/`COLORS`, чтобы работала смена темы; не хардкодить hex.
- Компоненты самодостаточны: тянут свои данные через React Query, ключи запросов стабильны.

## 14. Идемпотентность операций записи/импорта

Повторный запуск импорта/синка не должен дублировать данные: upsert по естественному ключу
или «очистить и залить заново». Пример — `importers/ttk_matrix.py`, синки в `scheduler.py`.

## 15. Не доверяй внешним данным — приводи на границе

Данные из внешних источников (iiko, Excel, API) валидируем и приводим к типам в одном месте
на входе: `_num()`/`_clean_unit()`/`_norm()` в импортёре, парсеры ответов iiko в клиенте.
Дальше по коду — уже чистые значения.

## 16. Доменные величины фиксируем явно

Единицы и смысл величины подписываем (per-portion vs per-batch, ₽ vs %, г vs мл). Урок проекта:
с/с ТТК `cost_total` — за весь выход рецепта (батч), а `cost_full` из «Сводной» — за порцию;
их нельзя путать при умножении на количество.

## 17. Внешние интеграции — таймауты и изоляция сбоев

У сетевых вызовов всегда таймаут. Некритичная интеграция не должна ронять основной сценарий:
ловим, логируем traceback, отдаём пусто/дефолт. Пример — `weather.py` (погода не валит дашборд).

---

> Правила можно менять: если правило мешает — обсуждаем и правим эту страницу.
