# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Что это

Система аналитики и учёта костов ресторана «Искенди» (storeId `161059`). Продажи тянутся из **внутреннего, нереверс-инженерного API iikoweb** (`https://iskendi.iikoweb.ru`) и отдаются фронту через FastAPI. Сверх дашборда ведётся **собственная БД** поставщиков, цен, тех-карт (ТТК) и себестоимости — **в iiko ничего не пишем, только читаем**. Справочник эндпоинтов/метрик iikoweb — в `IIKO_WEB_API.md`; дорожная карта — `PLAN.md`; журнал работ — `PROGRESS.md` (читать перед работой с API/планированием).

## Команды

Запуск в Docker (рекомендуется): `docker compose up -d --build`. Frontend на http://localhost:5173 (nginx, проксирует `/api` на backend), backend на http://localhost:8000. БД живёт в volume `backend-data`. Нужен заполненный `.env` в корне (его читает `env_file`); `DATABASE_URL` в контейнере переопределяется на `/data/iskendi.db`. Логи: `docker compose logs -f backend`.

Запуск локально без Docker: `./start.sh` (ставит зависимости, ставит chromium для Playwright, поднимает backend на :8000 и frontend на :5173).

Backend (из `backend/`):
- `pip install -r requirements.txt && python -m playwright install chromium`
- `uvicorn main:app --reload --port 8000`
- Триггер ручной синхронизации: `POST /api/sync`; здоровье: `GET /api/health`.
- Тестов нет.

Frontend (из `frontend/`):
- `npm install`, `npm run dev`, `npm run build` (`tsc -b && vite build`), `npm run lint`.

Код-стайл: см. `code-style.md` (корень). Backend приведён к нему (`black`/`isort`/`flake8`/`mypy`,
конфиг в `backend/pyproject.toml` и `backend/.flake8`, line-length 100). Перед коммитом:
`cd backend && black . && isort . && flake8 .`. Конфиг приложения — pydantic `Settings`
в `backend/config.py` (`from config import settings`), не `os.getenv` по коду.

Перед запуском backend нужен `.env` (см. `.env.example`): обязателен `IIKO_WEB_PASSWORD`, а также `AUTH_PASSWORD` (вход в дашборд; пустой пароль = вход невозможен).

## Архитектура

Поток данных: **iikoweb API → `IikoWebClient` → (а) `scheduler` синкает выручку по дням в SQLite, либо (б) роутеры дёргают iiko вживую (блюда, почасовые) → React-фронт**. Косты/ТТК/поставщики — из собственной БД (наполняется импортом xlsx).

### Авторизация в iikoweb (ключевой нюанс)
`backend/iiko_web_client.py` — внутренний API защищён cookie-сессией (TTL ~20 мин), отдельного login-endpoint нет. `_login()` поднимает **headless Chromium через Playwright**, вводит логин/пароль как человек на странице `/navigator/index.html#/auth/login`, забирает cookies и дальше шлёт быстрые JSON-запросы через `httpx` с этими куками. `_ensure_session()` проверяет сессию через `/api/auth` и перелогинивается при истечении; `_post()` при 401/403 сбрасывает куки и повторяет один раз. Пароль — только в `.env`.

### Авторизация дашборда (логин/пароль)
Весь дашборд закрыт за экраном входа. **Бэкенд** (`backend/auth.py`): один общий логин/пароль из `.env` (`AUTH_USERNAME`/`AUTH_PASSWORD`; пустой пароль = вход выключен, сверка в константное время через `hmac.compare_digest`). Сессия — **JWT (HS256), подписан вручную на `hmac`** (без внешних зависимостей), TTL `JWT_TTL_HOURS` (дефолт 12 ч); секрет подписи — `JWT_SECRET`, при пустом выводится из пароля (`iskendy:<auth_password>`). Роутер `backend/routers/auth.py`: `POST /api/auth/login` (логин/пароль → `{token, username}`), `GET /api/auth/me` (проверка токена). Зависимость **`require_auth`** (Bearer-JWT, иначе 401) навешена в `main.py` на **все роутеры и `/api/sync*`**; публичны только `/api/health` и `/api/auth/login`. **Фронт**: токен в localStorage (`src/token.ts`), axios-перехватчики (`src/api.ts`) добавляют `Authorization: Bearer` и при 401 чистят токен + редиректят на `/login`; guard `RequireAuth` (`src/auth.tsx`) оборачивает защищённые роуты в `App.tsx`; экран входа — `src/pages/Login.tsx`; кнопка «Выйти» — в `Sidebar`. Это **локальная авторизация дашборда**, не путать с cookie-сессией iikoweb выше.

### Единый эндпоинт данных
Почти все метрики идут через один вызов `POST /api/kpi/dashboard/get-data` (`get_metrics()`), параметризованный `metricCodes` + `dataType` (`DATA_SUMMARY_BY_DATE` для выручки по дням, `DATA_DETAILS` для блюд, `DATA_TOTAL` для итогов). Названия и категории блюд приходят **в том же ответе** в блоке `decoration.product` (с `with_decoration=True`) — отдельного справочника блюд не нужно. Коды метрик — выручка через `REV_GROSS` (бухгалтерские `ACC_CAT_*`/НДС в этой точке возвращают 0, не использовать).

### Синхронизация
`backend/scheduler.py` (APScheduler, таймзона `settings.timezone` = `Europe/Moscow`): `sync_revenue` и `sync_dishes` каждый час за 7 дней, `full_sync` (31 день) ночью в 00:05 и один раз при старте приложения (в `lifespan` в `main.py`). `RevenueDaily` обновляется по дню (upsert по PK-дате); `DishSale` за окно `(date_from, date_to)` полностью пере-вставляется (delete + insert). Каждый синк пишет результат в `SyncLog`. **`POST /api/sync`** (`main.py`): без параметра — `full_sync` (полный, кнопка «Синхронизировать»); `?days=N` — лёгкий синк за N дней (клиентская автосинхронизация по таймеру). Любой ручной синк **сбрасывает кэш** (`cache_clear()`). `GET /api/sync/last` — время последней успешной синхронизации.

### Кэш живых чтений
`backend/cache.py` — простой in-memory TTL-кэш (dict `key → (expiry, value)`). `get_metrics` и `olap_sales` в `iiko_web_client.py` кэшируют ответ по ключу из аргументов вызова (метрики/тип данных/даты/группировки) на `settings.cache_ttl_seconds` (`CACHE_TTL_SECONDS`, дефолт 60 c; 0 — выключить). Цель — схлопнуть повторные одинаковые живые запросы дашборда за один период. Сбрасывается `cache_clear()` при ручном синке (`POST /api/sync`), чтобы кнопка «Синхронизировать» всегда давала свежие данные.

### БД
`backend/models.py` — SQLAlchemy, SQLite. Таблицы: `revenue_daily`, `dish_sales`, `sync_log` (продажи) + `suppliers`, `supplier_contacts` (телефоны/каналы: phone/whatsapp/telegram/email, много на поставщика), `supplier_files`, `ingredients`, `supplier_prices` (есть поле `brand` — торговая марка предложения), `ttk`, `ttk_ingredients`, `dish_mappings` (привязка POS-имя↔ТТК) (косты/ТТК) + `daypart_plan` (план: дневная норма выручки/ср.чека/гостей на дейпарт × группу дня недели). `init_db()` создаёт схему (`create_all` идемпотентен — **новую таблицу добавляет без потери данных**); **миграций нет — при смене СУЩЕСТВУЮЩЕЙ схемы пересоздавать БД** (`docker compose down -v`). Файлы (накладные/прайсы) лежат в volume `/data/files` (`backend/storage.py`).

### Импорт ТТК/прайса
`backend/importers/ttk_matrix.py` парсит `backend/seed/ttk_matrix.xlsx` («ТТК и матрица продуктов», ~67 листов): «Матрица поставщиков» (прайс), «Матрица продуктов», «Сводная» (категории), лист-на-ТТК. **ТТК двухуровневые** (блюдо → п/ф → сырьё): строки резолвятся по нормализованному имени (`_norm` срезает «п/ф», ё→е, регистр), п/ф-ссылки предпочитают листы-полуфабрикаты. Канон-имя ТТК берётся из **имени листа** (B1 местами с копипаст-ошибкой). Разбор листа **независим от раскладки колонок** (`_col_map` ищет «с/с руб.»/«Выход» по шапке — у комбо они сдвинуты). **Порционная с/с (`cost_full`)**: строка «Итого с/с» листа (продукты+списания+упаковка) → «Сводная» → «Итого» рецепта (резерв для комбо); это надёжнее, чем неполная «Сводная». Идемпотентен (чистит ТТК/цены/ингредиенты, поставщиков апсёртит по `casefold`-ключу — SQLite `lower()` не трогает кириллицу!). **Канонизация + дедуп имён поставщиков**: алиасы (`_canon_supplier` + `SUPPLIER_ALIASES`, кросс-скрипт — WB/wildberries.ru → `wildberries.ru`; METRO/Мэтро/«METRO Chef …» → `METRO`; OZON/озон/ozon.ru → `Ozon`) + общий **ключ сопоставления `_supplier_key`** (lower, ё→е, срез `www.`/протокола/доменной зоны `.ru/.com`, только буквы/цифры — так «Turkish food»/«turkish-food»/«turkish-food.ru» = один). При импорте накопившиеся дубли **схлопываются**: контакты/файлы переносятся на «главного» (у кого их больше), имя приводится к канону, лишние строки удаляются. **Переживают реимпорт**: привязки `dish_mappings` (перелинковка по `name_norm` ТТК) и торговые марки `supplier_prices.brand` (снимок по поставщик+ингредиент). Запуск: `POST /api/import/ttk-matrix` или кнопка на странице Поставщики.

### API-слой
Коды метрик/типы данных/маппинги iiko вынесены в `backend/constants.py` (используй их, не хардкодь строки). **Переименование меню-категорий на дашборде** — `constants.CATEGORY_DISPLAY` + `utils.display_category()`: применяется единым выходным шагом во всех dishes-разрезах (`/api/dishes`, `check-composition`, `hourly-breakdown`, `basket`) и к подписям, и к ключам (чтобы drill/матч между виджетами не ломался; на детект доставки/модификаторов не влияет — те сверяют сырые имена). Сейчас: `«Меню» → «Допы и соусы»` (дефолтная корзина iiko с допами/соусами — Фри/Дип/Кетчуп/Перец). Роутеры в `backend/routers/`:
- `revenue.py` — `GET /api/revenue` (пресет → из SQLite; диапазон → живой; режим «Сумма»). В `summary.prev` — предыдущий сопоставимый период для **KPI-дельт** (`prev_period_range` в `utils.py`: **месяц/MTD сравнивается с тем же отрезком прошлого месяца**, день/неделя/диапазон — со скользящим окном той же длины; из БД, при отсутствии истории — живой запрос); в `prev_data` — выровненный по дню прошлый период для режима «Погода». К дням подмешивается погода (`weather.py`). `GET /api/revenue/hourly` — часы (`DATA_SUMMARY_BY_HOURS`). `GET /api/revenue/by-channel` и `/hourly-by-channel` — **выручка по дням/часам в разрезе каналов** (OLAP, хелпер `_channel_revenue`). `GET /api/revenue/by-weekday` — **свод по дням недели** (агрегат дней одного дня недели за период: кол-во дней, сумма/средняя выручка, чеки, ср. чек; источник тот же, что у `/api/revenue`). `GET /api/revenue/by-daypart` — **выручка/чеки/ср.чек по дейпартам** (Завтрак/Ланч/Полдник/Ужин/Ночь; границы в `DAYPARTS` (`constants.py`) из исторического «свода» ресторана): почасовые данные (`DATA_SUMMARY_BY_HOURS`) сворачиваются в окна, сумма сходится с `/api/revenue`. `GET /api/revenue/ops-report` — **ежедневный операционный отчёт** (аналог исторического Excel-свода «ОП»): дни периода (столбцы) × дейпарты (строки), по каждой клетке — выручка/чеки/гости/средний чек/**food cost %**. Источник — **один OLAP-запрос** по `[дата, час, OrderNum, категория, имя]` + `DishSumInt`/`DishAmountInt`/**`GuestNum`** (гости — атрибут заказа, берём раз на `OrderNum`); часы свёрнуты в окна `DAYPARTS`. Food cost = с/с порций (`_dish_unit_cost` из `dishes.py`, ТТК `cost_full`) ÷ выручка **блюд с привязкой** (+ поле `coverage` — какая доля выручки окна покрыта ТТК). Справа — `total` (Факт за период), `avg_per_day` (Среднее на активный день), `revenue_share` (доля дейпарта). Плюс **food cost % по группам категорий** (`category_groups`/`category_totals` + `categories` на дейпарт): группа категории через `_category_group` (`DRINK_CATEGORIES`→Напитки, `ALCOHOL_CATEGORIES`→Алкоголь — пуст у точки, остальное→Еда) — нижний блок свода. **Слой плана**: к каждому дейпарту и итогу подмешиваются `plan`/`plan_pct` (Факт/План/% к плану) из таблицы `daypart_plan` (роутер `plan.py`) — план задаётся **дневной нормой на (дейпарт × группа дня недели** `WEEKDAY_GROUPS`/`WEEKDAY_TO_GROUP`), на период масштабируется умножением на число подходящих дней (`_period_plan`, `group_day_count`) — поэтому `% к плану` честен для день/неделя/MTD/диапазона. `has_plan` — задан ли план. Скидок у точки нет (метрика = 0), поэтому `DishSumInt` численно совпадает с `REV_GROSS`. **Роутер `plan.py`** (`prefix=/api/plan`): `GET` — матрица норм, `PUT` — сохранить, `POST /seed-from-history?months=N` — автозаполнить средними дневными нормами из истории (OLAP). `GET /api/revenue/kpi-by-channel` — KPI в разрезе **доставка / не доставка** (брутто-суммы по каналам). **Параметр `include_delivery` (по умолчанию `true`)** для галки «без доставки»:
- В `revenue.py` (`/api/revenue`, `/api/revenue/hourly`, `/api/revenue/by-weekday`, `/api/revenue/by-daypart`): при `false` бэкенд вычитает выручку и чеки доставки (OLAP, хелперы `_delivery_buckets`/`_exclude_delivery`; доставка = меню-категория «Доставка», чек = заказ с такой позицией) из REV_GROSS-дней/часов — и из текущего, и из прошлого периода (чтобы KPI-дельты сравнивали сопоставимое). С/с по каналам не делится, поэтому `food_cost_pct`/`cost_sum` остаются по всей точке (в `by-weekday` food cost считается от полной выручки дня).
- **Что считается доставкой для галки** (`include_delivery=false`): **меню-категория «Доставка» ИЛИ имя позиции с маркером `_д`** (напр. «Дюрюм_д»). Единое правило — хелпер **`utils.is_delivery(category, name)`** (категория точным совпадением, `_д` — токеном в любом месте имени, регистронезависимо); используется во всех revenue/dishes-разрезах. Отсев **по-позиционно**, БЕЗ правила модификатора «Статус» и БЕЗ распространения на весь заказ. **`ChecksDistribution` теперь тоже управляется галкой** (`check-distribution`/`by-channel` принимают `include_delivery`): при выкл доставочные заказы/канал исключаются, плюс фронт **дополнительно скрывает строку «Доставка»** из разреза (фильтр в `ChecksDistribution.tsx`). В виджете **нет эмодзи** — метрики подписаны текстом «Чеки»/«Выручка». 3-way разрез на дашборде `Delivery` (`kpi-by-channel`/`service-breakdown`) по-прежнему опирается на «Статус» заказа и галкой не управляется.
- В `dishes.py` (`/api/dishes`, `/hourly-breakdown`, `/check-composition`, `/check-fullness`): при `false` отсекаются позиции категории «Доставка». `/api/dishes` просто фильтрует строки get-data (там уже есть категория) — без OLAP. OLAP-разрезы пропускают строки с `DishCategory == «Доставка»` (в `hourly-breakdown?group=dish` категория для этого добавляется в группировку). Заказ, целиком доставочный, после отсева исчезает сам (не остаётся товарных позиций).
- `dishes.py` — `GET /api/dishes` живой запрос за период; `group_by=dish|category`; **исключает `productType=MODIFIER`**; с/с per-portion = `Ttk.cost_full` × кол-во, **только не-п/ф** (коллизия имён п/ф↔блюдо); источник по блюду — ручная `DishMapping`, иначе авто-совпадение имени; поле `channel` = «доставка», если `is_delivery` (категория «Доставка» или имя с `_д`). **Флаг `has_cost`**: без ТТК-привязки `cost_sum=0`, а `cost_pct/margin_pct=null` (не выдаём мнимую 100% маржу; на фронте — «—»); для категории `has_cost` истинно, только если у **всех** её блюд есть привязка. `check-distribution` — чеки по типу обслуживания (**уникальные заказы** по каналу через OLAP `OrderNum`+«Статус»; сумма по типам = числу чеков, не кол-ву модификаторов). `hourly-breakdown?group=category|dish` — продажи по часам (блюда/категории, OLAP; в режиме `dish` каждый item несёт `category` — для drill-down «категория→блюда» на фронте). `basket` — **матрица сочетаемости** (market basket, #14): что чаще берут вместе в одном чеке (`group=category|dish`); возвращает `labels`/`matrix` (со-встречаемость пар в чеках) + `pairs` (топ-пары с `support`/`confidence`); OLAP по `OrderNum`. `service-breakdown` — блюдо×канал (OLAP по `OrderNum`+«Статус»). `check-composition` — средняя доля категорий в чеке (по кол-ву/выручке, период/часы). `check-fullness` — распределение чеков по числу позиций (1/2/3/4+) по часам, где **позиция = единица товара** (сумма `qty` по заказу, дробный вес округляется), а не число разных блюд. **Исключение модификаторов из OLAP-разрезов**: OLAP не отдаёт `productType`, поэтому хелпер `_modifier_filters` берёт имена+категории `productType=MODIFIER` из `dishes_detail` и отсекает их в `hourly-breakdown`/`service-breakdown`/`check-composition`/`check-fullness`/`basket` (модификаторы вроде «Разрезать 1/2» — не блюда). **Категорию отсекаем по `cats`, ТОЛЬКО если она ЦЕЛИКОМ модификаторная** (нет ни одного `DISH` — «модификаторы»/«Статус»): `cats = cat_has_mod − cat_has_dish`. **Смешанные категории НЕ отсекаются** (`Напитки`, `Доставка` содержат и блюда, и модификаторы-добавки) — иначе из всех OLAP-разрезов пропадала вся категория (баг: «Напитки» — 3-я по выручке — исчезала из состава чека, наполненности и т.д.). Отдельные модификаторы-строки в смешанной категории по имени не вычистить: платные «Кола»/«Айран» в OLAP **слиты** с одноимёнными блюдами (один `(name, category)`), отсев по имени убил бы реальные продажи; остаётся лишь бесплатный «Айран_» (rev=0) — пренебрежимо. «Статус» при этом сначала даёт канал, потом отсекается. `order-types` — диагностика поля OrderType (пуст у точки).
- `suppliers.py` — CRUD; **`DELETE /{id}`** — удаление поставщика (каскад контактов/цен/файлов + физические файлы через `storage.delete_file`; кнопка-корзина в списке Поставщики); `/{id}/contacts` (POST/PUT/DELETE, валидация `normalize_phone`/`normalize_email`); `/{id}/products/{price_id}` PUT — торговая марка; `/export` → xlsx (openpyxl); файлы (`UploadFile`/`FileResponse`).
- `nomenclature.py` — `prefix="/api"`, `/ingredients` (GET список/карточка + `PUT /{id}` правка), `/ttk` (каталог + карточки), `/dish-mappings` (GET/POST/DELETE).
- `imports.py` — запуск импорта.
Все роутеры с периодом принимают `period` **или** `date_from/date_to` (диапазон приоритетнее). `period_range` (`utils.py`): `day` = сегодня, **`week` = с понедельника текущей недели по сегодня** (не последние 7 дней; `prev_period_range` для недели сдвигает на 7 дней назад), **`month` = с 1-го числа текущего месяца по сегодня (MTD)**, не последние 30 дней. **«Сегодня» считается в часовом поясе ресторана** (`utils.today()` → `settings.timezone`, не TZ контейнера). `full_sync` синкает 31 день, чтобы MTD всегда был в БД. Производные метрики (food cost %, маржа, доли, дельты, с/с %) считаются **на лету в питоне**. Хелперы каналов/нормализации — в `utils.py`. `weather.py` — погода Москвы из Open-Meteo (кэш в памяти, не критична; **только дневная температура `temp_max`** — ночную `temp_min` не тянем и не показываем).

**OLAP-движок** (`iiko_web_client.olap_sales`) — для разрезов, которые `get-data` не умеет (блюдо×час, блюдо×канал, состав/наполненность чека). Поток: `POST /api/olap/init` → поллинг `fetch-status/{hash}` → `fetch/{hash}/simple` → `result.rows` (`field0` = склейка group-полей через «, », далее data-поля). **Тип обслуживания у точки НЕ в `OrderType` (пуст)** — берётся из модификатора категории «Статус» на уровне заказа: группировка по `OrderNum` связывает блюда заказа с его «Статусом». Поля/тип/вью и нетоварные категории — в `constants.py` (`OLAP_*`, `ORDER_STATUS_*`, `DELIVERY_CATEGORY`, `NON_PRODUCT_CATEGORIES`). Подробности полей — в `IIKO_WEB_API.md`. **dateTo в фильтре `date_range` по `OpenDate.Typed` — ВКЛЮЧИТЕЛЬНЫЙ целый день** (живой API игнорирует `includeRight=False`): передавать `dateTo = date_to` как есть, **без +1** — иначе в выборку попадёт лишний день после диапазона (для одного дня это удваивало все OLAP-метрики).

### Фронт
React 19 + Vite + TS, **react-router** (роуты в `App.tsx`, layout с `Sidebar`). Страницы в `src/pages/`: `Dashboard`, `Delivery` (отдельный дашборд доставки — пункт сайдбара `/delivery`), `Suppliers`, `SupplierCard`, `NewSupplier`, `Nomenclature`, `TtkCard`. **Дашборд разбит на 3 вкладки** (стейт `tab` в `Dashboard.tsx`, KPI-карточки всегда сверху): **Пульс** (RevenueChart + ChecksDistribution; **при одном дне** — `isSingleDay` в `Dashboard.tsx` — вместо вырожденного «по дням» показывается `HourlyChart` с внутридневной динамикой), **Операции** (WeekdaySummary + DaypartBreakdown + OpsReport + HourlyChart + CheckFullness + HourlyBreakdown; OpsReport скрыт при одном дне), **Меню** (CheckComposition + MenuBasket + MenuEngineering + DishTable). **Глубокий анализ каналов** (доставка / зал / с собой) живёт на отдельном дашборде `Delivery`; в Пульсе оставлена карточка `ChecksDistribution` — наглядный разрез чеков/выручки по типу обслуживания (**реагирует на галку «С доставкой»**: при выкл доставка исключается). Сверху — **галка «С доставкой»** (`withDelivery` в `Dashboard.tsx`, по умолчанию **выкл**, **без эмодзи**): выкл → **все виджеты дашборда** получают данные с `include_delivery=false` и показывают цифры **без доставки**. Рядом — кнопка «Синхронизировать» (полный синк) с подписью времени последней синхронизации (`fmtSync`: **сперва время, затем дата** — «14:05 · 23.06», МСК) и **селектор автосинхронизации** (`autoSyncMs`, опции `AUTOSYNC_OPTIONS` в `constants.ts`: выкл / 5 / 15 / 30 мин / 1 час; хранится в localStorage). При выбранном интервале `useEffect` ставит `setInterval`, который раз в интервал делает лёгкий синк (`triggerSync(AUTOSYNC_DAYS=7)`) + рефетч. Затрагивает revenue-виджеты (KpiCards, RevenueChart, HourlyChart, WeekdaySummary, DaypartBreakdown, OpsReport) и меню/блюда-виджеты (CheckFullness, HourlyBreakdown, CheckComposition, MenuEngineering, DishTable). Компоненты дашборда (что показывает каждый — в `README.md`): `KpiCards` (Выручка/Средний чек/Чеков с дельтами к пр. периоду; **все три из одного источника `REV_GROSS`** `/api/revenue` — сходятся между собой; разбивка по каналам убрана отсюда и вынесена на отдельный дашборд `Delivery`, чтобы не смешивать `REV_GROSS` и OLAP), `RevenueChart` («Выручка по дням»: режимы Сумма/По статусам (стек по каналам, `/api/revenue/by-channel`; **реагирует на галку «С доставкой»** — при выкл запрос идёт с `include_delivery=false` и канал «доставка» убирается из стека)/Погода, **форма всегда столбчатая** — переключатель типа графика убран как косметика (дни — дискретные корзины; линия/область рисовали бы ложную непрерывность и не несут раскраску по дням недели); фильтр дней — **скрыт при одном дне**; подпись дня на оси X — «Пн 22.06» (день недели + число в формате DD.MM, хелпер `dayLabel`/`dm`), под ней погода **структурно** (иконка · температура · дельта `(±°)` к тому же дню прошлого периода, цвет дельты: теплее `warn` / холоднее `accent`); в «Сумма» — один ряд выручки, **столбцы окрашены по группе дня недели** (`WEEKDAY_GROUPS` в `constants.ts`: Пн / Вт-Ср / Чт / выходные Пт-Вс) с легендой групп; **ср. чек и чеки не отдельными столбцами, а в кастомной подсказке** `SumTooltip`; опорная линия среднесуточной выручки), `WeekdaySummary` («Свод по дням недели»: строки окрашены по группе дня недели `WEEKDAY_GROUPS`, **фикс-порядок Пн→Вс** (сортировка/инлайн-бары убраны)), `DaypartBreakdown` («Выручка по дейпартам»: столбцы выручки по окнам дня Завтрак/Ланч/Полдник/Ужин/Ночь + доля над столбцом + таблица выручка/доля/чеки/ср.чек; источник `/api/revenue/by-daypart`, пустые окна скрыты), `OpsReport` («Ежедневный отчёт по дейпартам»: плотная таблица дни (столбцы) × дейпарты (строки), под каждым дейпартом подстроки Выручка/Ср.чек/Гости/Кост% (food cost раскрашен по `FOOD_COST_THRESHOLDS`), справа Факт/Среднее/Доля; ниже — отдельный блок **«Food cost % по дейпартам и категориям»** (дейпарт × Еда/Напитки/…, тултип с выручкой и покрытием ТТК); при заданном плане — колонки **План/% плана** (раскраска по выполнению) + кнопка **«⚙ План»** открывает `PlanEditor` (матрица норм дейпарт×группа дня, тумблер метрики, «Заполнить из истории», сохранение → инвалидация `ops-report`); шапка дней окрашена по `WEEKDAY_GROUPS`, левые 2 колонки липкие, **таблицы растянуты на всю ширину** (`minWidth: 100%` — при коротком периоде колонки дней заполняют ряд, при месяце — гориз. скролл); источник `/api/revenue/ops-report`; скрыт при одном дне), `HourlyChart` («Выручка по часам»: выручка + число чеков по часам), `ChecksDistribution` («Чеки и выручка по типу обслуживания»: доля чеков + доля выручки по каналу рядом, источники `check-distribution` + `revenue/by-channel`, матч каналов по lower-case), `CheckComposition` (состав чека; клик по категории → провал в товары, источник `/api/dishes`), `CheckFullness` (наполненность чеков + строка-инсайт: доля одиночных чеков и час-пик апсейла), `HourlyBreakdown` («Продажи по часам»: столбчатая диаграмма по часам **со стеком по категориям** (выручка/кол-во, тумблер Объём/Доли — стек 100%, клик по столбцу = выбор часа; диаграмма всегда `group=category`) + таблица блюд/категорий за час с инлайн-барами, сортировка; **в режиме «Категории» строки разворачиваются в блюда** этой категории за выбранный час — drill-down, грузит `group=dish` лениво), `MenuBasket` («Сочетаемость в чеке» — матрица сочетаемости #14: **крупная** тепловая матрица топ-позиций (ячейки 72×52), всегда по блюдам, источник `/api/dishes/basket`; **порядок осей фиксирован** тумблером «По популярности / А–Я» и при клике не двигается; **клик по подписи строки/столбца — подсветка** (focus): выбранное блюдо и всё, что с ним берут в чеках, **выделяются явно** — связанные подписи ярким текстом/жирным **на полупрозрачной акцентной подложке** (у выбранной — залитый чип, белый текст), ячейки крестовины с насыщенностью floor 0.45 + акцентная рамка (`boxShadow inset`), остальное гасится (opacity 0.12 + `grayscale`). **Порог видимости ячейки 0.5%** (`shown`): ниже — пустая, без числа/заливки/подсветки (иначе в режиме «Доля чеков» с мелкими значениями подсветка заливала бы пустые ячейки). helpers `linked` (связано ли блюдо с focus) / `cellOn` (в крестовине ли ячейка); повторный клик/«сбросить» — отмена; блок «Топ пар» убран), `MenuEngineering` (матрица популярность×маржа с залитыми квадрантами и подписями топ-6 блюд + ABC-Парето, источник `/api/dishes`; матрица и ABC-по-прибыли учитывают только блюда с `has_cost`), `DishTable` (группировка/сортировка/поиск; колонки **План с/с / Факт с/с / План кост % / Факт кост % / Δ кост / Маржа %** — «кост» = с/с÷цена реализации (=`cost_pct`); факт пока заглушка «—», появится после учёта накладных; при отсутствии с/с показывает «—», такие строки сортируются вниз), `Sidebar`. Почасовые графики (`HourlyChart`/`CheckFullness`/`HourlyBreakdown`) выравнивают ось X через `fillHourGaps` (`format.ts`) — часы без продаж между min/max не схлопываются. Выбор периода — пресет или диапазон (`RangeSel` в `api.ts`, хелперы `rangeQS`/`rangeKey`). Запросы — `src/api.ts`, типы — `src/types.ts`, форматтеры — `src/format.ts`, валидация (телефон/email) — `src/validation.ts`; **пороги/интервалы/палитра — `src/constants.ts`**.

**Раскладка:** `#root` без `max-width` — дашборд во всю ширину экрана. Виджеты вкладок раскладываются классом `.dash-grid` (`index.css`): 2 колонки на широких экранах, 1 при ширине < 1200px; `.dash-full` растягивает виджет на весь ряд (RevenueChart, ChecksDistribution, DaypartBreakdown, HourlyBreakdown, MenuEngineering, DishTable). Все карточки-графики используют `ResponsiveContainer`, поэтому корректно ужимаются в колонке.

**Темы:** палитра — CSS-переменные (`index.css`, `[data-theme]`), цвета через `var(--bg|card|grid|muted|text)` либо `COLORS` (тоже `var()`); акценты — фиксированные hex. Переключение — `src/theme.ts` (`useTheme`, хранит выбор в localStorage). Чтобы цвет реагировал на тему — используй переменные, а не hex. База API из `VITE_API_URL` (в Docker пусто → относительные пути через nginx).
