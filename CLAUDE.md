# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Что это

Система аналитики и учёта костов ресторана «Искенди» (storeId `161059`). Продажи тянутся из **внутреннего, нереверс-инженерного API iikoweb** (`https://iskendi.iikoweb.ru`) и отдаются фронту через FastAPI. Сверх дашборда ведётся **собственная БД** поставщиков, цен, тех-карт (ТТК) и себестоимости — **в iiko ничего не пишем, только читаем**. Справочник эндпоинтов/метрик iikoweb — в `IIKO_WEB_API.md`; дорожная карта — `PLAN.md`; журнал работ — `PROGRESS.md` (читать перед работой с API/планированием).

## Команды

Запуск в Docker (рекомендуется): `docker compose up -d --build`. Frontend на http://localhost:5173 (nginx, проксирует `/api` на backend), backend на http://localhost:8000. БД живёт в volume `backend-data`. Нужен заполненный `.env` в корне (его читает `env_file`); `DATABASE_URL` в контейнере переопределяется на `/data/iskendi.db`. Логи: `docker compose logs -f backend`.

Запуск локально без Docker: `./start.sh` (ставит зависимости, ставит chromium для Playwright, поднимает backend на :8000 и frontend на :5173).

Backend (из `backend/`):
- `pip install -r requirements.txt && python -m playwright install chromium`
- `uvicorn main:app --reload --port 8000`
- Триггер ручной синхронизации: `POST /api/sync`; здоровье: `GET /api/health`.
- Тестов нет.

Frontend (из `frontend/`):
- `npm install`, `npm run dev`, `npm run build` (`tsc -b && vite build`), `npm run lint`.

Код-стайл: см. `code-style.md` (корень). Backend приведён к нему (`black`/`isort`/`flake8`/`mypy`,
конфиг в `backend/pyproject.toml` и `backend/.flake8`, line-length 100). Перед коммитом:
`cd backend && black . && isort . && flake8 .`. Конфиг приложения — pydantic `Settings`
в `backend/config.py` (`from config import settings`), не `os.getenv` по коду.

Перед запуском backend нужен `.env` (см. `.env.example`): обязателен `IIKO_WEB_PASSWORD`.

## Архитектура

Поток данных: **iikoweb API → `IikoWebClient` → (а) `scheduler` синкает выручку по дням в SQLite, либо (б) роутеры дёргают iiko вживую (блюда, почасовые) → React-фронт**. Косты/ТТК/поставщики — из собственной БД (наполняется импортом xlsx).

### Авторизация в iikoweb (ключевой нюанс)
`backend/iiko_web_client.py` — внутренний API защищён cookie-сессией (TTL ~20 мин), отдельного login-endpoint нет. `_login()` поднимает **headless Chromium через Playwright**, вводит логин/пароль как человек на странице `/navigator/index.html#/auth/login`, забирает cookies и дальше шлёт быстрые JSON-запросы через `httpx` с этими куками. `_ensure_session()` проверяет сессию через `/api/auth` и перелогинивается при истечении; `_post()` при 401/403 сбрасывает куки и повторяет один раз. Пароль — только в `.env`.

### Единый эндпоинт данных
Почти все метрики идут через один вызов `POST /api/kpi/dashboard/get-data` (`get_metrics()`), параметризованный `metricCodes` + `dataType` (`DATA_SUMMARY_BY_DATE` для выручки по дням, `DATA_DETAILS` для блюд, `DATA_TOTAL` для итогов). Названия и категории блюд приходят **в том же ответе** в блоке `decoration.product` (с `with_decoration=True`) — отдельного справочника блюд не нужно. Коды метрик — выручка через `REV_GROSS` (бухгалтерские `ACC_CAT_*`/НДС в этой точке возвращают 0, не использовать).

### Синхронизация
`backend/scheduler.py` (APScheduler, таймзона `settings.timezone` = `Europe/Moscow`): `sync_revenue` и `sync_dishes` каждый час за 7 дней, `full_sync` (30 дней) ночью в 00:05 и один раз при старте приложения (в `lifespan` в `main.py`). `RevenueDaily` обновляется по дню (upsert по PK-дате); `DishSale` за окно `(date_from, date_to)` полностью пере-вставляется (delete + insert). Каждый синк пишет результат в `SyncLog`.

### БД
`backend/models.py` — SQLAlchemy, SQLite. Таблицы: `revenue_daily`, `dish_sales`, `sync_log` (продажи) + `suppliers`, `supplier_contacts` (телефоны/каналы: phone/whatsapp/telegram/email, много на поставщика), `supplier_files`, `ingredients`, `supplier_prices` (есть поле `brand` — торговая марка предложения), `ttk`, `ttk_ingredients`, `dish_mappings` (привязка POS-имя↔ТТК) (косты/ТТК). `init_db()` создаёт схему; **миграций нет — при смене схемы пересоздавать БД** (`docker compose down -v`). Файлы (накладные/прайсы) лежат в volume `/data/files` (`backend/storage.py`).

### Импорт ТТК/прайса
`backend/importers/ttk_matrix.py` парсит `backend/seed/ttk_matrix.xlsx` («ТТК и матрица продуктов», ~67 листов): «Матрица поставщиков» (прайс), «Матрица продуктов», «Сводная» (категории), лист-на-ТТК. **ТТК двухуровневые** (блюдо → п/ф → сырьё): строки резолвятся по нормализованному имени (`_norm` срезает «п/ф», ё→е, регистр), п/ф-ссылки предпочитают листы-полуфабрикаты. Канон-имя ТТК берётся из **имени листа** (B1 местами с копипаст-ошибкой). Разбор листа **независим от раскладки колонок** (`_col_map` ищет «с/с руб.»/«Выход» по шапке — у комбо они сдвинуты). **Порционная с/с (`cost_full`)**: строка «Итого с/с» листа (продукты+списания+упаковка) → «Сводная» → «Итого» рецепта (резерв для комбо); это надёжнее, чем неполная «Сводная». Идемпотентен (чистит ТТК/цены/ингредиенты, поставщиков апсёртит по `casefold`-ключу — SQLite `lower()` не трогает кириллицу!). **Переживают реимпорт**: привязки `dish_mappings` (перелинковка по `name_norm` ТТК) и торговые марки `supplier_prices.brand` (снимок по поставщик+ингредиент). Запуск: `POST /api/import/ttk-matrix` или кнопка на странице Поставщики.

### API-слой
Коды метрик/типы данных/маппинги iiko вынесены в `backend/constants.py` (используй их, не хардкодь строки). Роутеры в `backend/routers/`:
- `revenue.py` — `GET /api/revenue` (пресет → из SQLite; диапазон → живой; режим «Сумма»). В `summary.prev` — предыдущий сопоставимый период для **KPI-дельт** (`prev_period_range` в `utils.py`: **месяц/MTD сравнивается с тем же отрезком прошлого месяца**, день/неделя/диапазон — со скользящим окном той же длины; из БД, при отсутствии истории — живой запрос); в `prev_data` — выровненный по дню прошлый период для режима «Погода». К дням подмешивается погода (`weather.py`). `GET /api/revenue/hourly` — часы (`DATA_SUMMARY_BY_HOURS`). `GET /api/revenue/by-channel` и `/hourly-by-channel` — **выручка по дням/часам в разрезе каналов** (OLAP, хелпер `_channel_revenue`). `GET /api/revenue/by-weekday` — **свод по дням недели** (агрегат дней одного дня недели за период: кол-во дней, сумма/средняя выручка, чеки, ср. чек; источник тот же, что у `/api/revenue`). `GET /api/revenue/by-daypart` — **выручка/чеки/ср.чек по дейпартам** (Завтрак/Ланч/Полдник/Ужин/Ночь; границы в `DAYPARTS` (`constants.py`) из исторического «свода» ресторана): почасовые данные (`DATA_SUMMARY_BY_HOURS`) сворачиваются в окна, сумма сходится с `/api/revenue`. `GET /api/revenue/kpi-by-channel` — KPI в разрезе **доставка / не доставка** (брутто-суммы по каналам). **Параметр `include_delivery` (по умолчанию `true`)** для галки «без доставки»:
- В `revenue.py` (`/api/revenue`, `/api/revenue/hourly`, `/api/revenue/by-weekday`, `/api/revenue/by-daypart`): при `false` бэкенд вычитает выручку и чеки доставки (OLAP, хелперы `_delivery_buckets`/`_exclude_delivery`; доставка = меню-категория «Доставка», чек = заказ с такой позицией) из REV_GROSS-дней/часов — и из текущего, и из прошлого периода (чтобы KPI-дельты сравнивали сопоставимое). С/с по каналам не делится, поэтому `food_cost_pct`/`cost_sum` остаются по всей точке (в `by-weekday` food cost считается от полной выручки дня).
- **Что считается доставкой для галки** (`include_delivery=false`): **строго меню-категория «Доставка»** (в ней у точки продаются реальные блюда, поэтому заказ либо весь доставочный, либо весь нет). Отсев **по-позиционно по категории**, БЕЗ правила модификатора «Статус» и БЕЗ распространения на весь заказ. Это проще и уже, чем разрез каналов на дашборде `Delivery` (`kpi-by-channel`/`by-channel`/`service-breakdown`/`check-distribution`), который для 3-way зал/собой/доставка по-прежнему опирается на «Статус» заказа — эти виджеты галкой не управляются.
- В `dishes.py` (`/api/dishes`, `/hourly-breakdown`, `/check-composition`, `/check-fullness`): при `false` отсекаются позиции категории «Доставка». `/api/dishes` просто фильтрует строки get-data (там уже есть категория) — без OLAP. OLAP-разрезы пропускают строки с `DishCategory == «Доставка»` (в `hourly-breakdown?group=dish` категория для этого добавляется в группировку). Заказ, целиком доставочный, после отсева исчезает сам (не остаётся товарных позиций).
- `dishes.py` — `GET /api/dishes` живой запрос за период; `group_by=dish|category`; **исключает `productType=MODIFIER`**; с/с per-portion = `Ttk.cost_full` × кол-во, **только не-п/ф** (коллизия имён п/ф↔блюдо); источник по блюду — ручная `DishMapping`, иначе авто-совпадение имени; поле `channel` = «доставка», если категория == «Доставка». **Флаг `has_cost`**: без ТТК-привязки `cost_sum=0`, а `cost_pct/margin_pct=null` (не выдаём мнимую 100% маржу; на фронте — «—»); для категории `has_cost` истинно, только если у **всех** её блюд есть привязка. `check-distribution` — чеки по типу обслуживания (**уникальные заказы** по каналу через OLAP `OrderNum`+«Статус»; сумма по типам = числу чеков, не кол-ву модификаторов). `hourly-breakdown?group=category|dish` — продажи по часам (блюда/категории, OLAP). `service-breakdown` — блюдо×канал (OLAP по `OrderNum`+«Статус»). `check-composition` — средняя доля категорий в чеке (по кол-ву/выручке, период/часы). `check-fullness` — распределение чеков по числу позиций (1/2/3/4+) по часам, где **позиция = единица товара** (сумма `qty` по заказу, дробный вес округляется), а не число разных блюд. **Исключение модификаторов из OLAP-разрезов**: OLAP не отдаёт `productType`, поэтому хелпер `_modifier_filters` берёт имена+категории `productType=MODIFIER` из `dishes_detail` и отсекает их в `hourly-breakdown`/`service-breakdown`/`check-composition`/`check-fullness` (модификаторы вроде «Разрезать 1/2» — не блюда). «Статус» при этом сначала даёт канал, потом отсекается. `order-types` — диагностика поля OrderType (пуст у точки).
- `suppliers.py` — CRUD; `/{id}/contacts` (POST/PUT/DELETE, валидация `normalize_phone`/`normalize_email`); `/{id}/products/{price_id}` PUT — торговая марка; `/export` → xlsx (openpyxl); файлы (`UploadFile`/`FileResponse`).
- `nomenclature.py` — `prefix="/api"`, `/ingredients` (GET список/карточка + `PUT /{id}` правка), `/ttk` (каталог + карточки), `/dish-mappings` (GET/POST/DELETE).
- `imports.py` — запуск импорта.
Все роутеры с периодом принимают `period` **или** `date_from/date_to` (диапазон приоритетнее). `period_range` (`utils.py`): `day` = сегодня, `week` = последние 7 дней, **`month` = с 1-го числа текущего месяца по сегодня (MTD)**, не последние 30 дней. **«Сегодня» считается в часовом поясе ресторана** (`utils.today()` → `settings.timezone`, не TZ контейнера). `full_sync` синкает 31 день, чтобы MTD всегда был в БД. Производные метрики (food cost %, маржа, доли, дельты, с/с %) считаются **на лету в питоне**. Хелперы каналов/нормализации — в `utils.py`. `weather.py` — погода Москвы из Open-Meteo (кэш в памяти, не критична).

**OLAP-движок** (`iiko_web_client.olap_sales`) — для разрезов, которые `get-data` не умеет (блюдо×час, блюдо×канал, состав/наполненность чека). Поток: `POST /api/olap/init` → поллинг `fetch-status/{hash}` → `fetch/{hash}/simple` → `result.rows` (`field0` = склейка group-полей через «, », далее data-поля). **Тип обслуживания у точки НЕ в `OrderType` (пуст)** — берётся из модификатора категории «Статус» на уровне заказа: группировка по `OrderNum` связывает блюда заказа с его «Статусом». Поля/тип/вью и нетоварные категории — в `constants.py` (`OLAP_*`, `ORDER_STATUS_*`, `DELIVERY_CATEGORY`, `NON_PRODUCT_CATEGORIES`). Подробности полей — в `IIKO_WEB_API.md`. **dateTo в фильтре `date_range` по `OpenDate.Typed` — ВКЛЮЧИТЕЛЬНЫЙ целый день** (живой API игнорирует `includeRight=False`): передавать `dateTo = date_to` как есть, **без +1** — иначе в выборку попадёт лишний день после диапазона (для одного дня это удваивало все OLAP-метрики).

### Фронт
React 19 + Vite + TS, **react-router** (роуты в `App.tsx`, layout с `Sidebar`). Страницы в `src/pages/`: `Dashboard`, `Delivery` (отдельный дашборд доставки — пункт сайдбара `/delivery`), `Suppliers`, `SupplierCard`, `NewSupplier`, `Nomenclature`, `TtkCard`. **Дашборд разбит на 3 вкладки** (стейт `tab` в `Dashboard.tsx`, KPI-карточки всегда сверху): **Пульс** (RevenueChart + ChecksDistribution; **при одном дне** — `isSingleDay` в `Dashboard.tsx` — вместо вырожденного «по дням» показывается `HourlyChart` с внутридневной динамикой), **Операции** (WeekdaySummary + DaypartBreakdown + HourlyChart + CheckFullness + HourlyBreakdown), **Меню** (CheckComposition + MenuEngineering + DishTable). **Глубокий анализ каналов** (доставка / зал / с собой) живёт на отдельном дашборде `Delivery`; в Пульсе оставлена карточка `ChecksDistribution` — наглядный разрез чеков/выручки по типу обслуживания (она НЕ зависит от галки «С доставкой», всегда показывает все статусы). Сверху — **галка «С доставкой»** (`withDelivery` в `Dashboard.tsx`, по умолчанию **выкл**): выкл → **все виджеты дашборда** получают данные с `include_delivery=false` и показывают цифры **без доставки**. Затрагивает revenue-виджеты (KpiCards, RevenueChart, HourlyChart, WeekdaySummary, DaypartBreakdown) и меню/блюда-виджеты (CheckFullness, HourlyBreakdown, CheckComposition, MenuEngineering, DishTable). Компоненты дашборда (что показывает каждый — в `README.md`): `KpiCards` (Выручка/Средний чек/Чеков с дельтами к пр. периоду; **все три из одного источника `REV_GROSS`** `/api/revenue` — сходятся между собой; разбивка по каналам убрана отсюда и вынесена на отдельный дашборд `Delivery`, чтобы не смешивать `REV_GROSS` и OLAP), `RevenueChart` («Выручка по дням»: режимы Сумма/По статусам (стек по каналам, `/api/revenue/by-channel`)/Погода, **форма всегда столбчатая** — переключатель типа графика убран как косметика (дни — дискретные корзины; линия/область рисовали бы ложную непрерывность и не несут раскраску по дням недели); фильтр дней — **скрыт при одном дне**, погода+дельта на оси X; в «Сумма» — один ряд выручки, **столбцы окрашены по группе дня недели** (`WEEKDAY_GROUPS` в `constants.ts`: Пн / Вт-Ср / Чт / выходные Пт-Вс) с легендой групп; **ср. чек и чеки не отдельными столбцами, а в кастомной подсказке** `SumTooltip`; опорная линия среднесуточной выручки), `WeekdaySummary` («Свод по дням недели», сортировка, инлайн-бары в «Ср. за день»), `DaypartBreakdown` («Выручка по дейпартам»: столбцы выручки по окнам дня Завтрак/Ланч/Полдник/Ужин/Ночь + доля над столбцом + таблица выручка/доля/чеки/ср.чек; источник `/api/revenue/by-daypart`, пустые окна скрыты), `HourlyChart` («Выручка по часам»: выручка + число чеков по часам), `ChecksDistribution` («Чеки и выручка по типу обслуживания»: доля чеков + доля выручки по каналу рядом, источники `check-distribution` + `revenue/by-channel`, матч каналов по lower-case), `CheckComposition` (состав чека; клик по категории → провал в товары, источник `/api/dishes`), `CheckFullness` (наполненность чеков + строка-инсайт: доля одиночных чеков и час-пик апсейла), `HourlyBreakdown` («Продажи по часам»: столбчатая диаграмма по часам **со стеком по категориям** (выручка/кол-во, тумблер Объём/Доли — стек 100%, клик по столбцу = выбор часа; диаграмма всегда `group=category`) + таблица блюд/категорий за час с инлайн-барами, сортировка), `MenuEngineering` (матрица популярность×маржа с залитыми квадрантами и подписями топ-6 блюд + ABC-Парето, источник `/api/dishes`; матрица и ABC-по-прибыли учитывают только блюда с `has_cost`), `DishTable` (группировка/сортировка/поиск/с-с% — сам тянет данные; при отсутствии с/с показывает «—», такие строки сортируются вниз), `Sidebar`. Почасовые графики (`HourlyChart`/`CheckFullness`/`HourlyBreakdown`) выравнивают ось X через `fillHourGaps` (`format.ts`) — часы без продаж между min/max не схлопываются. Выбор периода — пресет или диапазон (`RangeSel` в `api.ts`, хелперы `rangeQS`/`rangeKey`). Запросы — `src/api.ts`, типы — `src/types.ts`, форматтеры — `src/format.ts`, валидация (телефон/email) — `src/validation.ts`; **пороги/интервалы/палитра — `src/constants.ts`**.

**Раскладка:** `#root` без `max-width` — дашборд во всю ширину экрана. Виджеты вкладок раскладываются классом `.dash-grid` (`index.css`): 2 колонки на широких экранах, 1 при ширине < 1200px; `.dash-full` растягивает виджет на весь ряд (RevenueChart, ChecksDistribution, HourlyBreakdown, MenuEngineering, DishTable). Все карточки-графики используют `ResponsiveContainer`, поэтому корректно ужимаются в колонке.

**Темы:** палитра — CSS-переменные (`index.css`, `[data-theme]`), цвета через `var(--bg|card|grid|muted|text)` либо `COLORS` (тоже `var()`); акценты — фиксированные hex. Переключение — `src/theme.ts` (`useTheme`, хранит выбор в localStorage). Чтобы цвет реагировал на тему — используй переменные, а не hex. База API из `VITE_API_URL` (в Docker пусто → относительные пути через nginx).
